# Ветки Публикации И Изменения

Читать этот файл после исследования источников, live-контура и единиц состояния.

Не выбирать ветку только по CMS. Выбирать по фактическому составу текущей операции.

## Общие Предусловия

До любой production-записи:

1. Зафиксировать желаемый результат и текущую команду пользователя.
2. Доказать live-домен и целевой Beget-контур.
3. Определить источники затронутых единиц.
4. Разделить deploy-owned и runtime-owned части.
5. Получить dry-run, manifest или эквивалентный diff.
6. Создать и проверить нужные бэкапы.
7. Подготовить откат и его триггеры.
8. Установить достаточность разрешения.
9. Запретить параллельное изменение выпуска на время операции.

## Ветка A. Статический Артефакт

Применять, если live состоит из готовых файлов и не хранит изменяемое состояние.

### Установить

- источник проекта;
- команду сборки;
- выходной каталог;
- состав артефакта;
- исключения `.git`, исходников, `node_modules`, секретов и черновиков;
- целевой путь;
- серверные redirects/headers;
- файловый откат.

### Порядок

1. Собрать и локально проверить артефакт.
2. Сравнить список файлов с текущим live.
3. Сделать файловый бэкап.
4. Показать добавления, замены и удаления.
5. Загрузить в подтверждённую папку.
6. Проверить фактические файлы и live URL.

Не считать локальный source-каталог готовым артефактом без проверки сборки.

## Ветка B. Серверный Код Без Базы

Применять, если сайт выполняет PHP или другой runtime, но не хранит рабочие данные в базе.

Дополнительно установить:

- web runtime и CLI runtime;
- расширения;
- server-owned config;
- writable каталоги;
- логи;
- кэш/сессии;
- синтаксическую и runtime-проверку.

Перед загрузкой проверять изменённый код совместимым CLI-бинарником. Не перезаписывать server-owned `.env`, локальные конфиги и writable runtime-каталоги.

## Ветка C. Файлы И База

Применять, когда код и данные образуют связанную версию.

### Построить Матрицу Совместимости

```text
новый код + старая база
старый код + новая база
новый код + новая база
```

Определить, какие комбинации допустимы и сколько времени.

### Выбрать Порядок

- сначала обратно совместимый код, затем миграция;
- сначала расширяющая миграция, затем код;
- maintenance window для неделимого изменения;
- staged switch для подготовленного контура.

Не выполнять разрушительную миграцию до доказанного отката. Для необратимой трансформации делать отдельную копию исходных данных и явно фиксировать ограничение.

### Бэкап

Получить согласованную точку файлов, базы и uploads. Если сайт продолжает принимать записи, использовать применимый транзакционный механизм, maintenance/read-only режим или другой доказанный способ.

### Откат

Если версии связаны, откатывать файлы и базу вместе. Не возвращать только код при несовместимой новой схеме.

## Ветка D. Управляемая CMS

Применять для CMS независимо от бренда.

Определить:

- CMS core, темы/шаблоны и плагины;
- content database;
- uploads/media;
- file-backed элементы;
- CMS-кэш;
- admin/manager;
- механизм обновлений;
- владельческие изменения после прошлого выпуска.

### Защищать

- uploads;
- конфигурацию подключения;
- пользовательский контент;
- аккаунты и роли;
- CMS-generated файлы, если они не являются disposable cache;
- плагины, установленные через CMS, если проект не управляет ими из Git.

### Проверять После Изменения

- публичную страницу;
- список/карточку затронутого типа;
- admin login;
- безопасное редактирование и сохранение;
- кэш;
- плагины/хуки;
- формы и фоновые действия.

Не запускать lifecycle-изменения из публичного запроса страницы. Использовать административное событие, CLI или закрытый импорт.

## Ветка E. Process-Based Приложение

Применять, если приложение требует процесса, порта, proxy, package manager, очереди или worker.

Определить:

- build artifact;
- runtime и package lock;
- server-owned env;
- команду старта;
- process manager;
- порт/proxy;
- health check;
- graceful restart;
- workers/queues/cron;
- writable данные и логи.

Порядок:

1. Собрать совместимый артефакт.
2. Подготовить зависимости без разрушения текущего процесса.
3. Проверить конфигурацию и свободные ресурсы.
4. Переключить или перезапустить согласованным способом.
5. Проверить health endpoint, процесс, логи и пользовательский маршрут.

Не использовать `kill` или произвольный restart до определения process manager и rollback.

## Ветка F. Неизвестный Или Смешанный Стек

Не писать в production. Вернуться к [stack-discovery.md](stack-discovery.md), пока не определены критические единицы, источники, runtime-owned границы, бэкап, откат и health checks.

После исследования собрать операцию из веток A-E. Смешанная система может требовать несколько веток как один согласованный выпуск.

## Первая Установка

Для новой системы дополнительно:

- проверить текущие требования runtime и базы;
- создать конфигурацию без попадания секретов в Git;
- ограничить bootstrap/setup поверхности;
- создать первого администратора безопасным способом;
- убрать установщики и временные инструменты;
- проверить права файлов;
- зафиксировать project contract;
- создать первый baseline-бэкап после принятой установки.

Не считать технический домен автоматически безопасным: закрыть индексацию, доступ или настроить дальнейший редирект по плану.

## Смена Папки Или Домена

Рассматривать как отдельную повышенно рискованную операцию.

Проверить:

- исходный и целевой контуры;
- абсолютные пути в конфигурации и базе;
- runtime-owned данные;
- site/base URL;
- domain binding;
- SSL;
- HTTP/HTTPS и `www`;
- canonical, robots и sitemap;
- технические дубли;
- DNS-период;
- откат привязки и данных.

Не удалять старый контур сразу. Сначала принять новый live, закрыть поисковый дубль и определить срок хранения.

## Перенос Между Аккаунтами Или Серверами

Дополнительно проверить:

- владельцев и права файлов;
- версии runtime, базы и расширений;
- абсолютные пути;
- квоты;
- отправку почты и внешние allowlist;
- DNS и период двойного трафика;
- доступ к старому серверу до нескольких проверенных backup generations.

## Файловая Синхронизация

До применения классифицировать каждый путь:

```text
deploy replace
deploy add
runtime preserve
server config preserve
generated cache rebuild
explicit delete
unknown — stop
```

Для удаляющей синхронизации обязательно:

- подтвердить абсолютный target;
- ограничить allowlist;
- получить dry-run;
- показать владельцу точный список удаления;
- сделать файловый бэкап;
- получить явное разрешение владельца на этот список;
- исключить uploads, config, backups и runtime data.

Общая команда на публикацию не разрешает неизвестное удаление. Если владелец уже одобрил точный список и после dry-run он не изменился, повторное подтверждение не требуется. Любое расширение списка останавливает запись.

Не строить shell-команду удаления из непроверенной строки пути.

## Временные Инструменты

Предпочитать CLI и каталоги вне webroot. Если публичный bootstrap неизбежен:

1. Ограничить доступ.
2. Свести действие к минимуму.
3. Не хранить секрет внутри файла дольше операции.
4. Удалить сразу после выполнения.
5. Подтвердить 404/403 на прежнем URL.

## Очистка Кэша И Производные Файлы

Очищать только после определения владельца кэша и влияния на live. После смены домена или контента пересоздать производные robots/sitemap/config только из фактических live-настроек и проверить выходной файл.

Не считать исходный шаблон доказательством сгенерированного результата.
