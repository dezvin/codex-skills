# Проверка Опубликованного Сайта

Читать этот файл после production-изменения и при read-only диагностике live.

Цель: доказать рабочий результат на фактическом домене. Upload, exit code 0 и HTTP 200 являются промежуточными признаками, а не полной приёмкой.

## Зафиксировать Baseline

До рискованной операции сохранить минимальное исходное состояние:

- канонический URL и redirects;
- главная;
- представители затронутых типов;
- административный вход;
- критическое пользовательское действие;
- robots, sitemap и 404;
- формы/cookies/аналитика при наличии;
- критичные server headers и логи.

Baseline нужен для сравнения и выбора отката.

## Соразмерный Охват

Для маленькой точечной правки проверить затронутую страницу, общий слой, критический маршрут и серверные ошибки.

Для структурного выпуска использовать три охвата:

### Полный Машинный Охват

Для всех известных публичных URL или полного затронутого набора:

- HTTP status;
- redirects;
- canonical и robots;
- внутренние ссылки;
- CSS/JS/images/fonts;
- сырые template/CMS маркеры;
- sitemap consistency.

### Представительный Визуальный Охват

- каждый затронутый шаблон/тип;
- desktop и mobile;
- длинный/короткий контент;
- пустое и максимальное состояние;
- общие header/footer/menu;
- 404.

### Полный Охват Критических Маршрутов

Пройти от входной страницы до результата пользователя: контакт, форма, покупка, вход, сохранение, поиск или другой целевой сценарий.

## Серверный Smoke

Проверить:

- ожидаемые файлы и права;
- runtime и расширения;
- соединение с ожидаемой базой;
- writable/cache каталоги;
- процесс и health check;
- cron/worker при затрагивании;
- свободное место;
- новые записи error log;
- отсутствие временных служебных файлов.

Не выводить секреты из конфигурации в отчёт.

## HTTP И Домен

Проверить фактический live:

- HTTP -> HTTPS;
- канонический `www`/без `www`;
- отсутствие redirect loops;
- SSL;
- главную и ключевые URL;
- старые URL после миграции;
- настоящий 404;
- CSS, JS, изображения и шрифты;
- cache headers там, где они ожидаются;
- отсутствие неожиданного 403/500/502/503.

Проверять тело и конечный URL, а не только первый status code.

## Браузер

В реальном браузере проверить:

- фактический текст и версию дизайна;
- header, navigation и footer;
- desktop, mobile и важную промежуточную ширину;
- изображения и layout;
- кнопки и ссылки;
- интерактив;
- console errors;
- network requests;
- отсутствие сырого CMS/template syntax;
- отсутствие визуального наложения и горизонтальной прокрутки.

## CMS/Admin

При наличии CMS проверить:

- вход в административный интерфейс;
- дерево/список данных;
- нужные элементы и плагины;
- безопасное обратимое сохранение, если выпуск затрагивал редактирование;
- публичный результат после сохранения;
- кэш;
- права владельца.

Вернуть тестовый объект в исходное состояние. Не тестировать опасный lifecycle на единственном реальном объекте.

## Данные И Пользовательские Действия

Проверить по области выпуска:

- целостность контента;
- количество и выборку сущностей;
- uploads;
- поиск, фильтры, pagination;
- формы и доставку сообщений;
- авторизацию и сессии;
- заказ/оплату без реальной транзакции, если нет согласованного тестового режима;
- интеграции и webhooks;
- фоновые действия.

Не отправлять реальные сообщения, платежи и массовые уведомления без явной необходимости и безопасного тестового режима.

## Cookies, Аналитика И Правовой Слой

Если затронуты общий шаблон, scripts, consent или домен:

- проверить состояние до согласия;
- проверить принятие/отказ по проектной логике;
- проверить network requests и cookies;
- убедиться, что аналитика не загружается раньше разрешённого состояния;
- проверить privacy/cookies/owner страницы;
- проверить footer links на mobile;
- не считать внешнее действие владельца выполненным кодом.

## SEO И Индексация

Проверить:

- title, description и H1;
- canonical на конечном домене;
- robots directives;
- robots.txt;
- sitemap URLs и status;
- redirects со старых адресов;
- отсутствие индексируемого технического дубля;
- отсутствие неожиданного `noindex`;
- корректность generated SEO-файлов после CLI/cron.

## Публичная Техническая Поверхность

Проверить недоступность:

- setup/install;
- import/sync/bootstrap;
- дампов и архивов;
- `.env` и конфигураций;
- Git и исходников;
- test/diagnostic файлов;
- публичных changelog/docs, если они раскрывают лишнее;
- технических доменов по принятому решению.

Ожидаемый status может быть 403, 404 или redirect согласно проекту. Важно отсутствие опасного содержимого.

## Производительность И Заголовки

Для структурного выпуска проверить:

- отсутствие резкого ухудшения загрузки;
- кеширование ассетов;
- сжатие;
- размеры изображений;
- базовые security headers;
- отсутствие лишнего `X-Powered-By`, если проект его скрывает;
- browser/server cache после изменения версии ассетов.

Лабораторный тест не заменяет реальные полевые данные. Не откатывать полезную функцию только ради одного балла без подтверждённой проблемы.

## Отделить Дефект От Шума

При timeout или сетевой ошибке:

1. Повторить запрос.
2. Проверить вторым инструментом.
3. Сравнить с серверным логом.
4. Проверить соседний URL.
5. Только затем классифицировать дефект сайта.

## Итоговый Статус

Использовать:

```text
PASS
PASS_WITH_ACCEPTED_RESIDUAL
ROLLBACK
BLOCKED
```

`PASS` — утверждённое изменение присутствует, критические проверки прошли, неожиданный drift отсутствует.

`PASS_WITH_ACCEPTED_RESIDUAL` — владелец понимает конкретный остаток, который не блокирует текущий результат.

`ROLLBACK` — сработал триггер и возвращена согласованная версия; откат также проверен.

`BLOCKED` — результат нельзя честно проверить или безопасно сохранить.
