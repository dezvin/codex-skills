# Исследование Неизвестного Стека

Читать этот файл, если технология неизвестна, смешана, плохо документирована или названию CMS нельзя доверять.

Цель: построить безопасную модель работающей системы без production-записи.

## 1. Прочитать Проект

Сначала read-only:

- применимые `AGENTS.md` и skills;
- структуру репозитория;
- Git-состояние;
- build/package/composer manifest;
- deploy-документы;
- конфигурационные шаблоны;
- CI/CD или release-команды;
- файлы исключений и generated/output каталоги.

Не считать наличие `index.html` доказательством статического сайта: он может быть частью сборки, заглушкой или кэшем.

## 2. Найти Точки Входа И Маркеры

Использовать маркеры как гипотезы:

| Признак | Возможный вывод | Что проверить |
|---|---|---|
| только готовые HTML/CSS/JS | статический артефакт | нет ли сборки/runtime/API |
| `composer.json`, PHP entry point | PHP-приложение | версия PHP, config, DB, cron |
| `wp-config.php`, `wp-content` | WordPress | DB, uploads, plugins, cache |
| `core/config`, `manager`, MODX bootstrap | MODX | DB, static elements, paths, cache |
| `package.json`, start script | Node/process app | build vs runtime, process, port, env |
| migrations/schema | приложение с БД | порядок миграций и откат |
| Docker-файлы | контейнерный контур | поддерживается ли он фактическим Beget-сервисом |

Не печатать содержимое файлов с секретами. Получать только безопасные признаки и идентификаторы.

## 3. Исследовать Фактический Live

Связать домен с серверным контуром по [beget-environment.md](beget-environment.md).

Установить:

- document root или entry point;
- runtime и версию;
- базу при наличии;
- владельцев/права файлов;
- процессы и cron;
- конфигурацию веб-сервера;
- кэш;
- uploads и server-side данные;
- внешние сервисы;
- текущие health checks.

Не изменять live для упрощения исследования.

## 4. Построить Карту Состояния

Для каждой найденной единицы заполнить:

```text
имя единицы
источник правды
deploy-owned или runtime-owned
кто изменяет
зависимости
что может потеряться
бэкап
проверка
откат
```

Примеры:

- тема может быть deploy-owned;
- CMS-контент и пользователи — runtime-owned в базе;
- uploads — runtime-owned на сервере;
- кэш — производное состояние, которое можно пересоздать;
- `.env` — server-owned конфигурация, а не deploy artifact;
- PHP — hosting-owned настройка.

## 5. Установить Направление Изменения

Спросить не «что лежит новее», а:

- какое состояние желаемое;
- какой источник владеет этой единицей;
- локальное должно заменить live или live нужно сохранить;
- есть ли пользовательские изменения после последнего выпуска;
- существует ли drift;
- можно ли применить изменение обратно совместимо.

Не использовать timestamp как единственное доказательство источника правды.

## 6. Найти Административный И Пользовательский Сценарий

Определить:

- как владелец редактирует контент;
- какой экран/команда подтверждает здоровье CMS;
- какой публичный маршрут критичен;
- какие формы или транзакции нельзя потерять;
- какие фоновые операции изменяют данные;
- какие интеграции должны продолжить работу.

## 7. Проверить Официальные Требования Стека

Если порядок зависит от CMS, runtime или базы, найти актуальную первичную документацию:

- требования версии;
- рекомендуемый upgrade/migration path;
- кэш и maintenance mode;
- структура uploads;
- backup/restore;
- database migrations;
- process restart.

Не строить ветку по памяти, если есть заметный шанс изменения требований.

## 8. Выбрать Класс

Выбрать минимально достаточный класс:

```text
A static artifact
B server code without persistent DB
C files + database
D managed CMS
E process-based application
F unknown/mixed — исследование не завершено
```

Название CMS не отменяет более общий класс. Например, CMS почти всегда требует ветку файлов + данных, но конкретный выпуск может менять только deploy-owned CSS.

## 9. Условия Остановки

Не переходить к production-записи, если неизвестно:

- какой сайт является целью;
- где его фактическая точка запуска;
- есть ли база или runtime-owned данные;
- какие каталоги нельзя перезаписывать;
- как получить согласованную копию;
- как откатить;
- как проверить административный и пользовательский результат.

Выдать точный список недостающих фактов. Не маскировать незнание готовой командой для «похожей» CMS.

## Выход Исследования

```text
технологический класс
доказанный live-контур
карта единиц состояния
источники и владельцы
deploy/runtime-owned граница
официальные требования стека
выбранная deployment branch
бэкап и откат
health checks
неизвестные, блокирующие запись
```
