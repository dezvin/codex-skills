# Бэкап, Откат И Инциденты

Читать этот файл перед рискованной записью, восстановлением, переносом, emergency hotfix и диагностикой серьёзной live-проблемы.

## Бэкап Как Точка Восстановления

Не считать бэкапом один файл с правильным именем. Доказать:

- принадлежность нужному домену/папке/базе;
- дату и логический момент;
- состав;
- ненулевой размер;
- читаемость архива/дампа;
- место вне webroot;
- права доступа;
- свободное место;
- известную процедуру восстановления.

## Выбрать Состав

Резервировать затронутые единицы:

- deploy-owned files;
- runtime-owned uploads;
- database;
- server config и веб-серверные правила;
- runtime/cron/process settings;
- domain binding/DNS values для переключения;
- внешний storage/export, если без него откат неполон.

Не копировать disposable cache как обязательную часть, если он гарантированно пересоздаётся. Но зафиксировать способ пересоздания.

## Согласованность Динамического Сайта

Файлы, база и uploads должны описывать совместимое состояние.

Выбрать один способ:

- maintenance/read-only окно;
- транзакционный snapshot базы плюс контролируемые uploads;
- приложение поддерживает обратно совместимое копирование;
- провайдерская согласованная копия;
- другой доказанный механизм конкретного стека.

Если согласованность не доказана, назвать остаточный риск до записи. Не выдавать набор разновременных копий за гарантированный полный restore point.

## Проверить Копии

### Файловый Архив

- архив существует;
- список содержимого читается;
- присутствуют ожидаемые ключевые пути;
- исключены вложенные бэкапы и временные секреты;
- архив не находится внутри копируемого webroot.

### Дамп Базы

- файл существует и не пуст;
- сжатие проверяется;
- присутствуют ожидаемые заголовки/таблицы;
- дамп завершился без проигнорированной ошибки;
- кодировка и формат совместимы с restore tool.

Не выводить строки с секретами или персональными данными в чат.

### Настройки

- зафиксировано предыдущее безопасное значение;
- известен способ возврата;
- для DNS записаны TTL и период распространения;
- для runtime известна рабочая предыдущая версия.

## Подготовить Откат До Записи

Записать:

```text
триггеры
какая версия возвращается
порядок файлов/базы/runtime
команды или действия восстановления
кэш после восстановления
live-проверки отката
ответственный внешний шаг
```

Если файлы и база связаны, определить единый release pair. Не смешивать «предыдущие файлы» с ещё более старой базой.

## Триггеры Отката

Определить до операции. Типичные примеры:

- live подключён не к той базе;
- массовые 500/502/503;
- недоступен admin/manager;
- потеря или массовое искажение данных;
- исчезли runtime-owned uploads;
- несовместимы код и schema;
- redirect loop или неверный канонический домен;
- критические формы/транзакции не работают;
- аналитика/consent работает в запрещённом состоянии;
- временный инструмент нельзя закрыть;
- критическая причина не устанавливается в допустимое окно.

Не выбирать откат только по второстепенному визуальному замечанию, если исправление безопасно и не нарушает окно. Не продолжать бесконечный fix-forward после критического триггера.

## Выполнить Откат

1. Остановить дальнейшие изменения и фоновые действия, которые ухудшат расхождение.
2. Подтвердить target ещё раз.
3. Вернуть связанную версию в определённом порядке.
4. Восстановить config/runtime/domain при затрагивании.
5. Очистить или пересоздать кэш.
6. Удалить временные инструменты.
7. Выполнить полный минимальный live-набор из [live-verification.md](live-verification.md).
8. Зафиксировать результат `ROLLBACK`, причину и оставшийся риск.

Rollback считается завершённым после проверки восстановленного live, а не после команды restore.

## Диагностировать Инцидент

Начинать read-only:

1. Зафиксировать время, симптом, URL и пользовательский эффект.
2. Проверить, что домен ведёт на ожидаемый контур.
3. Сравнить последние изменения проекта и сервера.
4. Проверить HTTP, браузер, процесс, runtime, место на диске и безопасные логи.
5. Проверить базу и внешние зависимости.
6. Отделить сетевой сбой от дефекта приложения.
7. Определить минимальное обратимое исправление или откат.

Не начинать с очистки всего кэша, перезапуска неизвестного процесса или замены папки: это уничтожает диагностические признаки и может расширить сбой.

## Emergency Hotfix

Применять, если ожидание обычного выпуска создаёт больший ущерб.

До записи:

- доказать target;
- определить источник затронутой единицы;
- сделать соразмерный бэкап;
- ограничить изменение минимумом;
- определить проверку и откат;
- установить достаточность текущего разрешения.

После стабилизации:

1. Перенести изменение в Git/CMS/config source.
2. Удалить временные диагностические средства.
3. Проверить drift между проектом и live.
4. Выполнить регрессию затронутого маршрута.
5. Записать только повторяемый вывод и существенную причину.

## Drift

Drift — расхождение между ожидаемым проектным состоянием и live.

Не считать любой drift ошибкой: CMS-контент и uploads могут законно жить только на live. Классифицировать:

```text
expected runtime state
approved server config
unbackported hotfix
manual unauthorized change
generated/disposable state
unknown — investigate
```

Исправлять направление по владельцу единицы:

- deploy-owned: вернуть live к проекту или backport подтверждённое live-изменение;
- runtime-owned: сохранить live и исключить из удаления;
- hosting-owned: обновить project contract после подтверждения;
- unknown: остановить destructive sync.

## Хранение И Уборка

До удаления определить срок хранения по риску выпуска, размеру и политике проекта.

Не удалять:

- единственную рабочую копию;
- предыдущий контур до приёмки нового;
- copy, нужную для незавершённого DNS-перехода;
- доказательные project notes без отдельной задачи.

Можно удалять только после показа точного списка и явного подтверждения владельца:

- временные credential-файлы;
- публичные bootstrap/diagnostic tools;
- временные staging-архивы после переноса в безопасное место;
- устаревшие копии по принятой retention policy.

Перед удалением повторно проверить абсолютный путь и не строить рекурсивную команду из непроверенной переменной. Общее разрешение на публикацию не заменяет решение по списку удаления; если уже одобренный список не изменился после dry-run, второй одинаковый вопрос не задаётся.

## Запись Об Инциденте

Фиксировать компактно:

```text
симптом и влияние
фактическая причина или UNKNOWN
что изменено/откачено
какая копия использована
как проверен live
какой drift устранён или остался
какое правило переносится на следующие операции
```

Не выдавать предположение о причине за подтверждённый факт.
