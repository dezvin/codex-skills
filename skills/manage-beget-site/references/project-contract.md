# Проектный Контракт Публикации

Читать этот файл при создании или обновлении project/deploy документа, манифеста выпуска и безопасной истории live-операций.

## Назначение

Глобальный skill хранит метод. Конкретный проект хранит только собственные проверенные факты, команды и границы.

Не навязывать новый файл поверх принятой структуры. Сначала искать:

- `AGENTS.md`;
- существующий `docs/deploy.md`;
- документы внутри `docs/<stack>/`;
- hosting/runbook файлы;
- project README, если он действительно является источником операций.

Для нового проекта рекомендуемое имя:

```text
docs/deploy.md
```

## Карта Источников

Не записывать одно поле «источник проекта». Описывать по единицам:

| Единица | Источник правды | Владелец изменения | Live-размещение | Бэкап | Откат |
|---|---|---|---|---|---|
| Код | `<repo/path>` | Codex/разработчик | `<target>` | Git + файлы | предыдущий выпуск |
| Контент | `<CMS/DB/file>` | владелец/Codex | `<DB/resource>` | DB/export | restore/revision |
| Uploads | `<server/storage>` | CMS/пользователь | `<path/bucket>` | файловая копия | restore |
| Runtime | `<Beget/panel>` | владелец/Codex | домен | фиксация настроек | обратное значение |

Адаптировать таблицу. Не добавлять несуществующие слои.

## Постоянные Сведения Проекта

Хранить без секретов:

```text
назначение проекта
live URL и канонический домен
технологический класс и известная CMS
источники правды по единицам
deploy-owned и runtime-owned части
команда сборки и выходной артефакт
SSH alias
подтверждённый абсолютный document root/entry point
web runtime и точная CLI-команда
идентификатор базы без пароля
uploads и изменяемые server-side данные
кэш и способ обновления
cron и процессы
серверные правила и редиректы
расположение бэкапов вне webroot
способ dry-run или предварительного diff
порядок публикации
триггеры и порядок отката
обязательные live URL и действия
известные внешние действия владельца
```

Не хранить:

- пароль Beget;
- приватный SSH-ключ;
- пароль базы;
- полный `.env`;
- дампы и архивы;
- временный credential-файл;
- личные данные без операционной необходимости;
- предположения, оформленные как проверенные пути.

## Статус Фактов

При неоднозначности использовать короткие статусы:

```text
verified — подтверждено проектом, сервером или live
reported — сообщено владельцем, но не проверено
inferred — выведено из нескольких признаков
unknown — нужно установить до операции
```

Не превращать deploy-документ в журнал размышлений. Оставлять статус только там, где он влияет на безопасность.

## Манифест Конкретной Операции

До production-записи сформировать:

```text
дата и идентификатор операции
сайт и live URL
текущая команда пользователя
операционный режим
технологический класс
источники затронутых единиц
deploy-owned и runtime-owned границы
доказанный целевой контур
точный состав изменения
явные исключения
точный видимый список удаления или `NO_DELETIONS`
dry-run/diff
бэкап и proof пригодности
согласованность точки восстановления
ожидаемая недоступность
триггеры отката
порядок отката
live-проверки
неизвестные и остаточный риск
достаточность текущего разрешения
```

Не обязательно сохранять полный манифест в репозитории для каждой мелкой операции. Он обязан существовать в рабочем контексте до записи.

## Достаточность Разрешения

Зафиксировать один из результатов:

```text
AUTHORIZED_BY_CURRENT_REQUEST
NEEDS_CONFIRMATION
READ_ONLY_ONLY
```

`AUTHORIZED_BY_CURRENT_REQUEST` допустим, если пользователь прямо поручил текущую live-операцию, назначение однозначно, а исследование не расширило состав и риск. При наличии удаления этот статус допустим только после того, как владелец увидел и явно одобрил точный список удаления.

`NEEDS_CONFIRMATION` использовать при новом разрушительном действии, неизвестном назначении, неожиданном diff, не показанном владельцу удалении или расширении на базу/домен/runtime-owned данные.

Общее `опубликуй` не считается разрешением неизвестного удаления. Если точный список уже одобрен и не изменился после dry-run, повторный вопрос не нужен.

Не хранить ритуальную подпись пользователя в файле. Важен смысл текущего поручения и соответствие манифесту.

## Запись О Выпуске

Для структурного или повторяемого выпуска сохранить:

```text
дата и идентификатор
что реально изменено
какой контур изменён
какие бэкапы созданы
какие live-проверки прошли
какие отклонения возникли
результат PASS / PASS_WITH_ACCEPTED_RESIDUAL / ROLLBACK
остаток и внешние действия
```

Не копировать длинный shell-лог. Фиксировать только то, что помогает повторить, проверить или откатить.

Для мелкой CMS-правки достаточно доказательства результата и короткого сообщения пользователю, если проект не требует журналирования.

## Emergency Hotfix И Drift

После прямой live-правки:

1. Установить соответствующий проектный источник.
2. Перенести туда изменение в текущем проходе.
3. Проверить, что следующий deploy не вернёт старое состояние.
4. Если backport невозможен, записать точное расхождение и ближайшее обязательное действие.

Не оставлять «сервер теперь правильный, локально потом» без долговременной фиксации.

## Обновлять Контракт

Обновлять документ, когда изменились:

- домен или document root;
- технология или способ запуска;
- runtime;
- база;
- источники правды;
- deploy/runtime-owned границы;
- способ сборки, публикации или отката;
- обязательные live-проверки;
- хранилище бэкапов.

Не обновлять постоянный контракт только потому, что изменилась одна статья или карточка товара.
