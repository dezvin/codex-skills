# Среда Beget

Читать этот файл при исследовании аккаунта, домена, серверной папки, PHP, базы, cron, процессов, SSL или способов доступа на Beget.

## Актуальность

Интерфейс панели, пути к бинарникам, доступные версии PHP и ограничения хостинга меняются. Перед рискованной или незнакомой операцией перепроверять текущую официальную справку Beget.

Полезные официальные точки входа:

- [управление сайтами и версиями PHP](https://beget.com/ru/kb/manual/sajty);
- [подключение по SSH из Windows](https://beget.com/ru/kb/how-to/ssh/kak-podklyuchitsya-po-ssh-iz-windows);
- [SSH-авторизация по ключу](https://beget.com/ru/kb/how-to/ssh/avtomaticheskaya-ssh-avtorizacziya-po-klyuchu);
- [полный бэкап через SSH](https://beget.com/ru/kb/how-to/other/polnyj-bekap-sajta-cherez-ssh);
- [перенос сайта между аккаунтами](https://beget.com/ru/kb/how-to/sites/perenos-sajta-s-akkaunta-na-akkaunt).

Не копировать команды из справки без подстановки фактического проекта и проверки текущего окружения.

## Доступ И Секреты

- Использовать уже настроенный SSH alias или безопасно настроить ключ.
- Не просить пароль Beget или приватный ключ в чате.
- Не коммитить ключи, пароли, дампы, архивы и полный `.env`.
- Не передавать пароль базы в аргументе команды, если он попадёт в историю или список процессов.
- Для временного credential-файла использовать каталог вне webroot, минимальные права и гарантированное удаление.
- Не печатать конфигурацию приложения целиком ради имени базы или хоста.

Проверять read-only доступ короткой командой:

```text
ssh -o BatchMode=yes <SSH_ALIAS> "pwd; id; hostname"
```

Не использовать команду как доказательство нужного сайта: она подтверждает только аккаунт и SSH-сессию.

## Найти Фактический Сайт

Не угадывать `public_html`. На одном аккаунте могут находиться несколько сайтов и несколько одноимённых папок.

1. Получить привязку домена к сайту из панели Beget или проверенной project/deploy фиксации.
2. Найти кандидаты серверных папок read-only способом.
3. Получить абсолютный путь через `readlink -f` или эквивалент.
4. Проверить характерные файлы приложения без вывода секретов.
5. Сопоставить безопасный уникальный признак с live-доменом.
6. Для динамического сайта связать папку с фактической базой и runtime.

Допустимый поиск кандидатов после подтверждения домашнего каталога:

```text
find <CONFIRMED_HOME> -maxdepth 4 -type d -name public_html -print
```

Ограничивать область и глубину. Не запускать тяжёлый поиск по всему серверу.

Цепочка доказательства:

```text
домен
-> Beget site binding
-> абсолютный document root или точка запуска
-> приложение
-> база/процесс при наличии
```

## PHP И Runtime

Не считать `php -v` в SSH доказательством версии, обслуживающей домен.

Проверять отдельно:

- web-PHP для конкретного домена в панели или безопасном диагностическом контуре;
- точный CLI-бинарник;
- нужные расширения;
- syntax check изменённых PHP-файлов тем же поколением PHP;
- runtime и команду запуска process-based приложения.

Фиксировать в проекте:

```text
web runtime
CLI runtime и абсолютная команда
ключевые расширения
способ проверить версию без публикации phpinfo
```

Не оставлять публичный `phpinfo()` или диагностический файл после проверки.

## Базы Данных

До изменения установить:

- фактическое имя базы без пароля;
- хост и тип базы;
- префикс таблиц, если он значим;
- приложение и домен, использующие базу;
- способ дампа и восстановления;
- права пользователя и хостинговые ограничения.

При `mysqldump` выбирать параметры по текущей версии и правам. Если хостинг запрещает отдельные операции, использовать подтверждённый совместимый режим, а не игнорировать ошибку.

Для изменяемого сайта определить согласованность:

- остановить записи на короткое окно;
- использовать транзакционный дамп, если он применим;
- либо зафиксировать, почему файлы/uploads и база остаются совместимыми без остановки.

## Домены, SSL И DNS

До изменения определить, какой слой управляет каждым правилом:

- панель Beget;
- `.htaccess` или конфигурация веб-сервера;
- приложение/CMS;
- внешний DNS-провайдер;
- CDN/proxy.

Не дублировать один редирект в нескольких слоях без необходимости. После изменения проверить:

- HTTP -> HTTPS;
- канонический вариант `www`/без `www`;
- отсутствие циклов;
- действующий сертификат и цепочку;
- канонический домен внутри приложения;
- период DNS-перехода, когда пользователи могут попадать на разные серверы.

DNS, привязка домена и SSL являются production-записью. Проверять достаточность разрешения до изменения.

## Cron И Процессы

Для process-based проекта установить:

- фактическую команду запуска;
- рабочий каталог;
- runtime;
- переменные окружения;
- способ безопасного перезапуска;
- health check;
- лог процесса;
- cron и фоновые задания, способные изменить данные во время миграции.

Не перезапускать неизвестный процесс как диагностический первый шаг.

## Бэкапы На Beget

Можно использовать:

- собственный архив/дамп выпуска;
- подтверждённую копию панели Beget;
- оба уровня для рискованного переноса.

Провайдерская копия не освобождает от проверки:

- нужного времени;
- состава файлов и базы;
- срока хранения;
- способа восстановления;
- принадлежности нужному сайту.

Хранить проектные архивы вне webroot. Проверять квоту перед созданием крупной копии.

## Минимальный Результат Исследования

```text
SSH alias и аккаунт без секрета
live-домен
подтверждённый document root/entry point
технологический класс
web runtime и CLI runtime
база при наличии
runtime-owned каталоги и данные
доменные/SSL правила
cron/процессы при наличии
место бэкапов
health check
```

Если критичный факт нельзя получить read-only способом, остановить production-запись и назвать точное недостающее подтверждение.
