## Когда использовать

Использовать, когда JTBD-файл уже существует, а новые фразы, интервью,
наблюдения, возражения или ситуации могут уточнить его содержание.

Цель режима — не переписать анализ заново, а понять реальное влияние новых
данных и предложить минимальные обоснованные изменения.

## Необходимый вход

1. Точный путь или содержимое текущего JTBD-файла.
2. Один или несколько новых сигналов с доступным контекстом.
3. Цель обновления, если она уже названа.

Прочитать текущий файл целиком, если его размер позволяет это без потери
рабочего контекста. Для очень большого файла сначала прочитать структуру, затем
релевантные разделы и связанные определения.

## Порядок

1. Сохранить каждый сигнал в исходной формулировке и отделить его от
   интерпретаций.
2. Разобрать сигнал по `mode-client-signal.md`.
3. Найти в текущем JTBD утверждения и разделы, которых сигнал касается.
4. Сопоставить новое наблюдение с текущей моделью.
5. Классифицировать влияние:
   - **подтверждает:** усиливает существующий вывод без изменения смысла;
   - **уточняет:** делает контекст, формулировку или границу точнее;
   - **добавляет:** показывает новую materially distinct job, ситуацию, силу,
     категорию или сегментную особенность;
   - **противоречит:** не согласуется с текущим утверждением;
   - **снижает уверенность:** показывает, что текущая формулировка слишком
     сильная или широкая;
   - **сегментно:** верно только для части аудитории;
   - **пока не влияет:** данных недостаточно или случай нерелевантен модели.
6. Прочитать reference только того этапа, чья логика затронута:
   - jobs и triggers -> `stage-2-jobs.md`;
   - альтернативы -> `stage-3-competition.md`;
   - боли, критерии, возражения и другие категории ->
     `stage-4-deep-analysis.md`;
   - сегменты -> `stage-5-segmentation.md`;
   - Push / Pull / Anxiety / Habit -> `stage-6-forces.md`;
   - практические действия -> `stage-7-recommendations.md`.
7. Проверить последствия изменения для связанных разделов. Не менять их
   автоматически, но показать рассогласования.
8. Подготовить предложение, а не сразу редактировать файл.

## Правило силы сигнала

Не использовать механический порог количества наблюдений. Оценивать:

- прямоту и конкретность источника;
- наличие контекста найма;
- повторяемость в независимых источниках;
- согласованность с поведением, а не только словами;
- наличие альтернативных объяснений;
- масштаб изменения, которое предлагается внести.

Один сильный сигнал может исправить явную фактическую ошибку или добавить
помеченную гипотезу. Один неоднозначный сигнал не должен превращаться в общее
утверждение обо всей аудитории.

## Предложение изменения

Для каждого затронутого места показать:

```text
Раздел текущего JTBD
Текущая формулировка
Новый сигнал и источник
Тип влияния
Что предлагается изменить
Почему это изменение уместно
Статус доказательности и уверенность
Какие связанные разделы затрагиваются
Что останется неизменным
```

Если изменение пока рано вносить, предложить не правку канона, а проверяемую
гипотезу или вопрос для следующего исследования.

## Подтверждение и запись

Перед записью:

1. Показать все предлагаемые изменения и противоречия.
2. Назвать точный путь файла.
3. Перечислить точные разделы, которые будут изменены.
4. Задать один явный вопрос о принятии результата и записи этого объема.
5. Остановиться и дождаться ответа.

После подтверждения внести минимальную связную правку. Сохранить полезную
исходную структуру и не переписывать соседние разделы ради стиля. Добавить
достаточную прослеживаемость источника без помещения личных данных в канон.

Если пользователь принимает только часть предложений, записать только ее.
Непринятые гипотезы не смешивать с утвержденной моделью.

## Границы

- Не считать сам факт предоставления файла разрешением на его изменение.
- Не добавлять сигнал в конец файла без смысловой интеграции.
- Не превращать частный случай в свойство всей аудитории.
- Не удалять противоречащие данные ради гладкой модели.
- Не пересобирать весь JTBD, если достаточно точечной правки.
- Не обновлять связанные продуктовые, коммерческие или контентные документы
  без их отдельного решения и маршрута.
