## Когда нужен рабочий файл

Использовать рабочий файл для полного анализа, длинной частичной работы или
пакета материалов, который нельзя надежно удержать в одном диалоге.

Не создавать его по умолчанию для одного клиентского сигнала. Создать только
если пользователь хочет накапливать сигналы как отдельное исследование.

## Создание

После утверждения рамки Этапа 1:

1. Предложить понятный путь отдельного исследовательского Markdown-файла.
2. Показать, какие разделы будут созданы.
3. Получить явное разрешение на запись.
4. Не использовать `jtbd.md` или другой канонический файл как рабочий черновик.

Рабочий файл хранит текущее состояние анализа, а не историю переписки и не
служебный журнал агента.

## Структура рабочего файла

```markdown
# JTBD-анализ: [название]

## Состояние
- Текущий этап:
- Статус этапа:
- Последнее подтвержденное решение:
- Следующее действие:

## Рамка
- Продукт / услуга:
- Цель анализа:
- Рабочие предварительные сегменты:
- Основные допущения:

## Источники и доказательства
- Использованные материалы:
- Ключевые дословные сигналы:
- Ограничения данных:

## Raw JTBD по сегментам

## Consolidated JTBD

## Карта сегмент -> работа

## Конкурентный контекст

## Детализация JTBD

## Сегментация

## Forces of Progress

## Рекомендации

## Открытые гипотезы и противоречия

## Принятые решения
```

Не заполнять будущие разделы общими заглушками. Пустой заголовок допустим до
начала соответствующего этапа.

## Обновление после этапа

На каждом контрольном вопросе:

1. Показать результат этапа.
2. Назвать точные разделы рабочего файла, которые будут обновлены.
3. Объединить вопрос о принятии этапа и записи показанного результата.
4. После явного подтверждения записать только принятый результат.
5. Обновить текущий этап, последнее решение и следующее действие.

Не хранить скрытые рассуждения, отладочные заметки или дословную переписку.
Сохранять только данные, выводы, допущения, решения, ограничения и
прослеживаемость.

На Этапе 4 фиксировать каждую завершенную job и каждый утвержденный блок
категорий отдельно. Это позволяет сохранить полноту без удержания всей матрицы
в одном ответе.

## Продолжение

При возобновлении:

1. Прочитать рабочий файл.
2. Определить последний утвержденный результат и текущий незавершенный этап.
3. Проверить, нет ли новых данных или решения пользователя, меняющих рамку.
4. Прочитать `core.md`, этот файл и reference только текущего этапа.
5. Кратко сообщить пользователю, что уже принято, что остается гипотезой и
   какой ближайший шаг продолжается.
6. Не повторять завершенные этапы без новой причины.

Если состояние неполно или противоречиво, восстановить только подтверждаемую
часть и явно назвать пробел. Не додумывать утраченное решение.

## Финал

Рабочий файл не обязан становиться итоговым отчетом. После Этапа 7 собрать
отдельный финальный документ по `final-report.md`. Перенос выводов из отчета в
канонические владельцы остается отдельным решением.
