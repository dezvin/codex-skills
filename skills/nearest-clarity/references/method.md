# Метод ближайшей ясности

Статус: рабочая версия v0.6

Дата: 2026-06-12

## 0. Назначение

Метод ближайшей ясности нужен для сложных задач, где нельзя честно составить точный план до конца, но и нельзя действовать наугад.

Метод помогает LLM-партнёру и человеку вести работу так, чтобы после каждой итерации становилось понятнее:

- какой целевой контур удерживаем;
- что уже достаточно ясно;
- что пока остаётся гипотезой;
- какой ближайший участок можно планировать;
- какой сигнал дала последняя итерация;
- нужно ли обновить карту решений.

Метод применим к разным задачам: запуск продукта, контент, курс, система, исследование, личное планирование, пересборка проекта, сложное решение.

Метод не заменяет предметную экспертизу. Он задаёт способ работы с неопределённостью, зависимостями, гипотезами и новыми сигналами.

## 1. Короткая формула

```text
Держим целевой контур.
Намечаем карту решений.
Планируем ближайший участок.
Делаем одну итерацию.
Заранее называем ожидаемый сигнал.
Считываем сигнал.
Обновляем модель ситуации.
```

Иначе:

```text
Не планировать туман как будто он уже ясен.
Но и не действовать вслепую.
```

## 2. Основная проблема

В сложной работе есть две частые ошибки.

Первая - слишком жёсткий план. Человек или LLM расписывает весь путь заранее, хотя часть вводных ещё неизвестна. Такой план создаёт ложную уверенность и часто ломается при первом столкновении с реальностью.

Вторая - движение только ближайшим действием. Это выглядит гибко, но без общей карты легко превращается в блуждание: каждый отдельный шаг разумный, а общий путь не складывается.

Метод ближайшей ясности соединяет три слоя:

- целевой контур - чтобы понимать, ради чего идёт работа;
- рабочий план ближайшего участка - чтобы не расплываться;
- короткие итерации - чтобы учитывать новые сигналы.

## 3. Когда использовать метод

Использовать метод, если:

- работа должна продолжаться короткими ограниченными итерациями;
- пользователь не понимает, что делать дальше среди нескольких
  жизнеспособных направлений;
- важные данные или ограничения появятся только по ходу выполнения;
- текущий план или основание работы могли устареть;
- нужно выбрать ближайший проверяемый участок, не выдавая дальний путь за
  известный;
- весь маршрут нельзя честно спланировать заранее.

Не использовать метод только потому, что задача сложная, стратегическая,
рискованная или многошаговая, если маршрут и критерий готовности уже ясны. Не
использовать его для обычной приоритизации по заданным критериям.

Если нужен одноразовый пересмотр рамки с рекомендацией или планом, который
завершается за один проход, использовать самостоятельный `zoom-out`, а не
`nearest-clarity`.

## 4. Режимы глубины

Метод не всегда нужно включать полностью.

### 4.1. Лёгкий режим

Использовать для небольших задач с умеренной неопределённостью.

Достаточно ответить:

```text
Цель.
Ограничение.
Ближайшее действие.
Ожидаемый результат.
```

### 4.2. Рабочий режим

Использовать для обычных сложных задач.

Нужно пройти:

```text
целевой контур -> карта решений -> ближайший участок -> итерация -> сигнал -> следующий шаг
```

### 4.3. Глубокий режим

Использовать для стратегических, рискованных или дорогих решений.

Добавить:

- явные допущения;
- карту ограничений;
- риски;
- развилки;
- критерии успеха;
- точку согласования перед исполнением.

Если задача длинная, дорогая или многоходовая, LLM-партнёр может предложить вспомогательные списки:

- журнал сигналов;
- список гипотез;
- карту решений со статусами.

Это не обязательные документы, а опциональные рабочие опоры. Их не нужно создавать, если задача удерживается без них.

## 5. Термины метода

### Целевой контур

Рамка, которая отвечает на вопрос:

```text
Ради чего ведётся работа и что нельзя потерять?
```

Целевой контур фиксирует смысл, желаемый результат, границы и грубые ошибки выбора.

### Карта решений

Предварительная карта возможного пути.

Она показывает:

- вероятные этапы;
- развилки;
- риски;
- зависимости;
- решения, которые пока рано принимать;
- места, где нужна проверка.

Карта решений не является обещанием маршрута. Она обновляется по новым сигналам.

### Ближайший участок

Часть работы, которую уже можно планировать достаточно конкретно.

Ближайший участок отвечает:

- что делаем в текущей фазе;
- в каком порядке;
- почему именно так;
- какой результат должен появиться;
- что нельзя делать раньше времени.

### Итерация

Один короткий проход работы: действие или небольшой набор действий, после которых должен появиться результат или сигнал.

Перед итерацией нужно назвать ожидаемый сигнал: что именно должно стать понятнее после действия.

### Сигнал

То, что стало понятнее после итерации.

Сигналом может быть:

- факт;
- реакция;
- ограничение;
- противоречие;
- новая гипотеза;
- подтверждение;
- отказ от старой версии;
- новая зависимость;
- риск.

Не каждый шум является сигналом. Один случай не должен автоматически становиться правилом.

### Zoom out

Одноразовый обзор с выводом, рекомендацией или планом является самостоятельным
`zoom-out`.

Внутри активного `nearest-clarity` Zoom out является зависимой операцией: он
пересобирает рамку, когда новое знание меняет основание работы, после чего
управление возвращается в итерационный процесс.

### Тестовая итерация

Маленькая проба на реальном материале, если анализ не даёт достаточной ясности.

Тестовая итерация дорогая и используется редко: только после обоснования и явного согласия человека.

## 6. Градиент точности

План не должен быть одинаково подробным на всём расстоянии.

```text
текущая итерация -> максимально конкретно
ближайший участок -> рабочий план
дальше -> карта решений
самый дальний горизонт -> целевой контур
```

Ошибка - требовать от дальнего горизонта точности текущей итерации.

Другая ошибка - вообще не смотреть дальше ближайшего действия.

Дальние элементы нужно маркировать честно:

- **ориентир** - общий целевой контур;
- **возможный этап** - скорее всего понадобится, но форма может измениться;
- **развилка** - есть несколько путей, решение позже;
- **риск** - место, где может обнаружиться проблема;
- **обзорная проверка** - нужно подняться уровнем выше перед следующим решением.

## 7. Рабочий алгоритм

### 7.1. Перед началом участка

1. Определить, что мы пытаемся сделать.
2. Задать целевой контур.
3. Назвать входные данные.
4. Назвать ограничения.
5. Наметить карту решений.
6. Выделить ближайший участок.
7. Определить результат ближайшего участка.
8. Выбрать первую итерацию.

### 7.2. Во время итерации

1. Сформулировать фокус.
2. Назвать основание: данные, документы, наблюдения, вводные.
3. Назвать ожидаемый сигнал: что должно стать понятнее после действия.
4. Выполнить действие.
5. Считать фактический сигнал.
6. Отделить факт от гипотезы.
7. Проверить, не появилось ли новое знание, требующее Zoom out.
8. Назначить следующий шаг.
9. Зафиксировать статус результата.

### 7.3. После итерации

Ответить:

```text
Что сделали?
Какой сигнал ожидали?
Что стало понятнее?
Что осталось неизвестным?
Какой сигнал получен?
Что меняется в карте решений?
Какой следующий шаг?
Какой статус результата?
```

Минимальные статусы знания:

- **факт** - подтверждённая входная информация;
- **наблюдение** - замеченное проявление, которое ещё не стало правилом;
- **гипотеза** - проверяемое предположение;
- **решение** - выбранный следующий шаг или рамка действия;
- **риск** - место возможной ошибки или потери;
- **шум** - информация, которая не меняет решение;
- **проверенный результат** - вывод, который выдержал проверку и может использоваться дальше.

## 8. Работа с гипотезами

Гипотеза - это проверяемое предположение, а не решение.

Формат:

```text
Мы предполагаем, что...
Потому что...
Это важно, потому что...
Проверить можно через...
Если гипотеза верна, следующий шаг...
Если неверна, меняем...
```

Неожиданный хороший результат тоже не становится правилом сразу.

Сначала он получает статус гипотезы:

```text
Это сработало в этом случае.
Пока не ясно, почему.
Нужно проверить, повторяется ли это и при каких условиях.
```

Запрещено подгонять объяснение под удобную версию.

Если причина не ясна, нужно назвать несколько возможных объяснений и уровень уверенности.

## 9. Новое знание и Zoom out

Если появилось новое знание, которое меняет основание работы, нельзя просто продолжать по старому плану.

Сначала нужен Zoom out.

При первоначальном запуске `nearest-clarity` зависимый `zoom-out` нужен только
если пользователь явно просит пересобрать рамку или текущее основание
подтверждённо устарело. Сам по себе запрос следующего шага не требует
`zoom-out`.

Короткий Zoom out:

1. Какой это тип задачи?
2. Какая цель выше уровнем?
3. В какой системе живёт задача?
4. Какие входные данные и ограничения изменились?
5. Что является фактом, а что предположением?
6. Какие зависимости изменились?
7. Какие риски и слепые пятна появились?
8. Как теперь выглядит критерий хорошего результата?
9. Что нужно обновить: целевой контур, карту решений, ближайший участок или текущую итерацию?

Формула:

```text
Новое знание -> Zoom out -> обновление карты решений -> новая итерация.
```

Если новое знание не меняет основание работы, его можно учесть в текущем цикле без полной пересборки.

Zoom out считается завершённым только если после него стало ясно, что именно меняется: цель, ограничения, зависимости, риски, критерий результата, карта решений, ближайший участок или текущая итерация.

Если после Zoom out ничего не изменилось и следующий шаг не стал яснее, это был не рабочий пересмотр, а лишний обзор.

## 9.1. Сигнал и шум

Сигнал - это новая информация, которая меняет понимание задачи, ограничения, гипотезы, риски, порядок действий или критерий результата.

Шум - это информация, которая выглядит заметно, но не меняет решение.

Сигналом может быть: повторяющаяся реакция, сильное противоречие, новое ограничение, ошибка в основании, подтверждение гипотезы, провал гипотезы, появление более сильного пути.

Шумом может быть: единичная случайная реакция, эмоциональный всплеск без связи с решением, красивая идея без применения, новый факт без влияния на текущий участок.

Правило:

```text
Не каждый новый факт требует пересборки карты.
Но каждый факт, который меняет основание решения, требует Zoom out.
```

Перед обновлением карты решений нужно спросить:

- что именно изменилось;
- на какой уровень это влияет;
- это повторяемый сигнал или одиночный шум;
- какое решение стало другим из-за этой информации.

## 10. Ограничения и точки согласования

Перед рискованным действием нужно назвать ограничения:

- что нельзя нарушить;
- какие ресурсы ограничены;
- какие решения необратимы;
- где нужна проверка;
- где нужно согласие человека.

Точка согласования нужна, если действие меняет:

- документ;
- план;
- деньги;
- публичную коммуникацию;
- направление проекта;
- обязательства;
- необратимое решение.

Схема:

```text
анализ -> предложение -> явное согласие -> исполнение
```

Для маленьких обратимых действий этот цикл не нужно раздувать.

Для рискованных переходов его нельзя пропускать.

## 10.1. Матрица переходов

Эта матрица помогает выбрать следующий режим работы.

| Ситуация | Что делать |
|---|---|
| Итерация дала понятный сигнал, карта решений не изменилась | Продолжать ближайший участок |
| Итерация дала сигнал, который меняет порядок действий | Обновить рабочий план ближайшего участка |
| Появилось новое знание, которое меняет цель, ограничения или основание задачи | Сделать Zoom out |
| Сигнал слабый, одиночный или не влияет на решение | Зафиксировать как наблюдение, не перестраивать карту |
| Ближайший участок дал нужный результат | Закрыть участок и выбрать следующий |
| Участок не даёт сигнала после нескольких итераций | Сузить задачу или сделать Zoom out |
| Действие меняет документ, план, деньги, публичную коммуникацию или направление проекта | Запросить точку согласования |
| Анализа недостаточно для выбора следующего шага, но маленькая проба дешевле большого ошибочного плана | Предложить тестовую итерацию с обоснованием и ждать согласия |

## 11. Старый материал и старые планы

Если старый элемент не работает целиком, его не нужно сразу выбрасывать.

Сначала проверить:

- что в нём умерло;
- что устарело;
- что всё ещё полезно;
- можно ли извлечь живую часть;
- можно ли использовать как мост;
- нужно ли отложить до другого этапа.

Формула:

```text
Сначала извлечь живое.
Потом удалить мёртвое.
```

Удалять стоит только после проверки, что в элементе нечего сохранить.

## 12. Тестовая итерация

Иногда проект нельзя понять только из анализа. Тогда может понадобиться маленькая тестовая итерация на реальном материале.

Этот режим можно неформально называть “разведкой боем”, но рабочий термин - тестовая итерация.

Тестовая итерация не является стандартным действием метода.

Она допускается только если:

- обычный анализ не даёт достаточной ясности;
- без тестовой итерации нельзя выбрать следующий шаг;
- цена маленькой пробы ниже цены большого ошибочного плана;
- LLM-партнёр явно предлагает этот режим;
- LLM-партнёр объясняет, почему без него нельзя обойтись;
- человек явно одобряет тестовую итерацию.

Формула:

```text
Тестовая итерация - только по обоснованному запросу и после явного согласия.
```

Если тестовая итерация дала полезный результат, его нельзя сразу превращать в стандарт.

Правильная цепочка:

```text
тестовая итерация -> наблюдение -> гипотеза -> проверка -> правило
```

Неправильная цепочка:

```text
тестовая итерация -> сразу новый стандарт
```

## 13. Защита от ложного прогресса

Ложный прогресс:

- есть большой список тем, но нет архитектуры;
- есть тексты, но нет пути читателя;
- есть план, но неизвестно, какие гипотезы он проверяет;
- есть красивый документ, но он не помогает принять решение;
- есть обсуждение, но следующий шаг не стал яснее;
- есть скорость, но нет проверки направления;
- есть итерации, но нет фиксации сигналов;
- есть Zoom out, но нет действия после него.

Проверка:

```text
После этой итерации стало легче решить, что делать дальше?
```

Если нет, это активность, а не продвижение.

Хорошая итерация не обязана дать приятный результат. Она обязана дать рабочую ясность: подтвердить гипотезу, ослабить гипотезу, показать риск, уточнить ограничение, закрыть развилку или честно показать, что данных недостаточно.

Главный критерий метода:

```text
Стало ли легче принять следующее правильное решение?
```

## 14. Частые поломки метода для LLM

LLM должна избегать этих ошибок:

1. **Слишком ранний финальный план**
   Делать вид, что весь путь уже понятен.

2. **План без статусов**
   Не различать факт, гипотезу, решение и проверенный результат.

3. **Итерации без сигнала**
   Выполнять действия без ожидаемого сигнала и без понимания, что должно стать яснее.

4. **Zoom out как ритуал**
   Делать обзор ради вида, без обновления рамки, карты решений или следующего шага.

5. **Тестовая итерация как побег от решения**
   Предлагать пробу там, где уже достаточно данных для следующего шага.

6. **Удаление старого материала без разбора**
   Выбрасывать старый план или текст, не извлекая живые части.

7. **Терминологический шум**
   Использовать метод как набор красивых слов, а не как рабочий алгоритм.

8. **Самовольное исполнение рискованных действий**
   Менять документы, планы, публичную коммуникацию или направление проекта без точки согласования.

## 15. Когда фиксировать результат

Фиксировать нужно не каждую мысль, а только то, что должно пережить текущий разговор.

Фиксировать стоит:

- принятое решение;
- рабочую рамку;
- сильную гипотезу;
- следующий шаг;
- ограничение;
- критерий;
- ошибку, которую нельзя повторить;
- структуру, которую будем использовать дальше.

Не фиксировать:

- сырой шум;
- случайные идеи без статуса;
- промежуточные эмоции;
- списки “можно бы когда-нибудь” без следующего действия.

## 16. Чеклист применения

Перед началом:

```text
1. Что мы пытаемся сделать?
2. Зачем это нужно?
3. Какой целевой контур?
4. Какие входные данные есть?
5. Какие ограничения есть?
6. Что видно на карте решений?
7. Какой ближайший участок?
8. Какая итерация даст больше ясности?
9. Какой сигнал мы ожидаем от первой итерации?
```

После итерации:

```text
1. Что сделали?
2. Какой сигнал ожидали?
3. Какой сигнал получили?
4. Что стало фактом?
5. Что осталось гипотезой?
6. Нужно ли делать Zoom out?
7. Что меняется в карте решений?
8. Какой следующий шаг?
9. Какой статус результата?
```

## 17. Критерий хорошего прохода

Проход считается хорошим, если после него есть:

- один конкретный результат;
- заранее названный ожидаемый сигнал;
- ясный сигнал;
- честный статус результата;
- обновлённая карта решений, если она изменилась;
- понятный следующий шаг;
- меньше тумана, чем до прохода.

Не обязательно получить финальный ответ.

Достаточно честно сузить неопределённость.

## 18. Критерий завершения ближайшего участка

Ближайший участок можно закрывать, если:

- получен результат, ради которого участок запускался;
- основные сигналы считаны;
- ключевые гипотезы получили статус;
- карта решений обновлена;
- понятно, какой следующий участок открывается;
- нет незакрытого рискованного решения без согласования.

Если эти условия не выполнены, участок не закрыт. Нужно продолжить итерации, сузить задачу или сделать Zoom out.

## 20. Интеграция с TODO.md

Метод ближайшей ясности может работать вместе с проектной task-системой, если в проекте установлен `TODO.md`.

Связь простая:

`Метод ближайшей ясности управляет движением. TODO.md фиксирует только то, что нужно не потерять.`

Метод не заменяет `TODO.md`.

`TODO.md` не должен становиться журналом сигналов, картой решений или стенограммой применения метода.

Использование `TODO.md` как источника информации не делает
`manage-project-tasks` владельцем процесса. Если нужно выбрать направление при
настоящей неопределённости, родителем остаётся `nearest-clarity`.
`manage-project-tasks` подключается только для отдельной операции сохранения
или восстановления долговременного task state.

### 20.1. Что попадает в TODO.md

Если задача должна пережить текущую сессию, её можно зафиксировать в `TODO.md`.

Для сложной задачи поля `TODO.md` используются так:

- `Goal` - целевой контур задачи: что должно стать правдой и что нельзя потерять.
- `Current Action` - текущая ближайшая итерация или рабочий шаг.
- `Items` - текущая и оставшаяся работа, а не вся история итераций.
- `Progress` - краткий итог уже полученного значимого сигнала или закрытого участка, максимум 1-3 пункта.
- `Constraints` - ограничения, которые реально влияют на следующий шаг.
- `Risks` - риски, которые могут изменить порядок действий или критерий результата.
- `Open Questions` - вопросы, без которых нельзя уверенно продолжить.
- `Decisions` - принятые решения, которые нужно помнить в будущей сессии.
- `Proof` - минимальное подтверждение текущего состояния или результата.
- `Next` - следующий полезный шаг.

### 20.2. Что не попадает в TODO.md

Не записывать в `TODO.md`:

- полный ход рассуждений;
- подробную карту решений;
- журнал всех сигналов;
- все гипотезы подряд;
- сырой Zoom out;
- промежуточный шум;
- дальний план, который ещё не может быть честно детализирован;
- эмоциональные реакции;
- случайные идеи без решения превратить их в задачу.

Если работа требует отдельной карты решений, списка гипотез, журнала сигналов или плана участка, Codex может предложить отдельный рабочий документ.

Создавать такой документ можно только если:

- без него трудно продолжать работу;
- задача длинная, рискованная или многоходовая;
- нужно передать состояние между сессиями;
- пользователь явно попросил зафиксировать такой слой.

### 20.3. Как метод влияет на статусы задач

Метод не добавляет новые task-статусы.

Если проектная task-система использует статусы `pending`, `in_progress`, `blocked`, `done`, `cancelled`, метод не меняет их.

Статусы знания метода - факт, наблюдение, гипотеза, решение, риск, шум, проверенный результат - не являются task-статусами.

Их можно кратко упоминать внутри `Progress`, `Risks`, `Open Questions` или `Decisions`, если это помогает продолжить задачу.

Пример:

`Signal: first test showed that the route works for one real case, but this is still an observation, not a rule.`

Не создавать в `TODO.md` отдельные статусы вроде:

- `hypothesis`;
- `validated`;
- `signal_received`;
- `zoom_out_needed`;
- `needs_clarity`.

Если нужна проверка, создать отдельную задачу проверки, а не новый статус.

### 20.4. Когда включать TODO.md при работе по методу

Сохранять результат через установленную task-систему, если:

- задача должна продолжиться в будущей сессии;
- появился следующий шаг, который нельзя потерять;
- есть блокер;
- нужна отдельная будущая проверка;
- принято решение, влияющее на дальнейшую работу;
- сложная задача разрослась и стала проектной задачей;
- пользователь явно попросил записать задачу.

Не создавать и не изменять task state, если:

- это короткая разовая итерация внутри текущего разговора;
- результат не должен пережить сессию;
- обсуждение пока не стало задачей;
- появилась сырая идея без цели и следующего действия;
- метод используется только как лёгкая рамка ответа.

Простое чтение `TODO.md` как входных данных выполняется напрямую и само по
себе не включает `manage-project-tasks`.

### 20.5. Closure и архив

Если задача, ведущаяся по методу, закрывается, закрытие происходит по правилам установленной task-системы.

Для `Codex_TODO_System` это означает:

- закрытая задача получает `done` или `cancelled`;
- результат, proof и follow-up уходят в `archive/tasks/YYYY-MM.md`;
- закрытая задача удаляется из `TODO.md`;
- если нужен следующий участок, создаётся новая `pending` задача.

В архив не нужно переносить весь методологический ход.

Архивная запись должна отвечать:

- что было закрыто;
- какой результат получен;
- чем это подтверждено;
- есть ли follow-up.

Если в проекте установлен `Codex_TODO_System`, использовать его правила closure-pass и archive-pass как источник истины для закрытия задач. Этот метод не должен вводить собственный порядок закрытия задач поверх task-системы.

### 20.6. Минимальная запись сложной задачи в TODO.md

Компактная запись сложной задачи должна содержать:

- название задачи;
- статус;
- цель;
- текущую итерацию или рабочий шаг;
- краткий прогресс, если он нужен;
- текущие и оставшиеся пункты;
- критерий готовности;
- минимальный proof;
- следующий шаг.

Не превращать такую запись в дневник применения метода.

Плохая запись выглядит так:

- записан весь ход рассуждений;
- перечислены все промежуточные сигналы;
- сохранён сырой Zoom out;
- перечислены все гипотезы без отбора;
- дальний горизонт расписан как точный план;
- нет ясного следующего действия.

### 20.7. Короткое правило интеграции

```text
Простая задача -> обычный ответ или короткий план.
Одноразовый пересмотр рамки -> самостоятельный zoom-out.
Продолжительная работа при настоящей неопределённости -> метод ближайшей ясности.
TODO.md как источник информации -> прямое чтение релевантных секций.
Сохранение долговременного состояния -> task-система через manage-project-tasks.
Закрытый результат -> архив задач.
Сырой ход мысли -> не фиксировать.
```

## 21. Короткая версия для вставки в работу

```text
Работаем по методу ближайшей ясности.

Держим целевой контур: зачем ведём работу и что нельзя потерять.
Намечаем карту решений: возможные этапы, развилки, зависимости и риски, не выдавая её за точный план.
Планируем ближайший участок: несколько зависимых шагов, которые уже можно пройти осмысленно.
Дальше идём короткими итерациями: фокус, основание, ожидаемый сигнал, действие, фактический сигнал, решение, статус.
Если новые данные меняют основание работы, сначала делаем Zoom out: уточняем цель, систему, факты, допущения, зависимости, риски и критерий результата. Потом обновляем карту решений.
Статус результата не завышаем: факт, наблюдение, гипотеза, решение, риск, шум и проверенный результат - разные вещи.
Тестовую итерацию используем редко: только если её необходимость обоснована и человек явно согласился.
Рискованные действия выполняем только через точку согласования: анализ, предложение, явное согласие, исполнение.
```
