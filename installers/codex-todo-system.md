# Codex TODO System

## 0. Назначение

Этот документ описывает переносимую систему ведения `TODO.md` для проектов, где работает Codex.

Документ предназначен для прямого чтения Codex. Его можно передать в новый проект и попросить: «Установи». После этого Codex должен настроить короткий корневой контракт, проектный skill `manage-project-tasks`, `TODO.md` и ленивый архив закрытых задач. Тяжёлая канбан-система и дополнительные task-протоколы для этого не нужны.

Главная цель: дать проекту простой, устойчивый и проверяемый слой задач, который переживает отдельные сессии и не теряет контроль, когда проект разрастается.

## 1. Основная идея

`TODO.md` — это проектный реестр открытой работы.

Он отвечает на вопросы:

- к какому внутреннему проекту относится задача;
- к какой части системы, функционалу или направлению относится задача;
- зачем это направление существует;
- что нужно сделать;
- что нужно сделать позже;
- что нельзя забыть;
- что сейчас в работе;
- что заблокировано;
- какое следующее действие;
- чем подтверждается текущее состояние задачи.

`TODO.md` не является:

- общим файлом инструкций проекта;
- README;
- журналом всего диалога;
- постоянным архивом завершённых задач;
- архитектурным документом;
- хранилищем сырых идей без решения превратить их в задачи;
- заменой истории Codex.

`TODO.md` нужен не для всего подряд, а для незакрытых задач, которые должны сохраняться между сессиями, относятся к плану, должны быть выполнены позже или требуют понятного статуса.

Для нового проекта закрытая история по умолчанию хранится отдельно:

`archive/tasks/YYYY-MM.md`

Если проект уже использует другой действующий путь, его нужно сохранить и объявить в корневом `AGENTS.md`. Во всех пояснительных примерах ниже `archive/tasks/YYYY-MM.md` означает путь по умолчанию, а не требование перенести существующий архив.

Архив — это компактная запись задач со статусом `done` или `cancelled`, а не журнал всех действий агента.

## 2. Главная структура TODO.md

`TODO.md` должен быть организован по смысловым областям, а не просто как плоский список задач.

Даже если проект сейчас маленький, структура всё равно должна поддерживать рост проекта. Не упрощать её только потому, что на старте задач мало.

Базовая структура:

```md
# TODO.md

## Project: <project-name>

### Purpose

- <why this internal project exists>

### Area: <area-name>

#### Purpose

- <why this area exists>

#### Active

<current or near-current tasks>

#### Later

<future tasks to remember>
```

Смысл уровней:

- `Project` — крупный внутренний проект или рабочий контур.
- `Area` — часть системы, функциональный блок, направление или подпроект.
- `Purpose` у проекта — зачем существует крупный контур.
- `Purpose` у области — зачем нужна эта часть системы.
- `Goal` у задачи — что должно стать правдой после выполнения конкретной задачи.

Пример структуры должен быть обезличенным и шаблонным. Не использовать в эталонной структуре реальные названия пилотных проектов, исторические данные или задачи из установки.

Допустимый пример:

```md
## Project: Project 1

### Purpose

- Explain why Project 1 exists.

### Area: Area 1

#### Purpose

- Explain why Area 1 exists inside Project 1.

#### Active

##### Task 1

Status: in_progress
Created: YYYY-MM-DD
Updated: YYYY-MM-DD

###### Goal

- Describe what should become true after Task 1 is completed.

###### Current Action

- Describe what Codex is doing now.

###### Items

- [~] in_progress: Work on the current meaningful step.

###### Done When

- Define the acceptance condition.

###### Proof

- Add minimal evidence available while the task remains open.

###### Next

- Define the next action.
```

## 3. Главное правило включения TODO.md

Количество шагов не является главным критерием.

Даже задача из одного действия должна быть записана в `TODO.md`, если пользователь явно говорит или подразумевает:

- «сделать позже»;
- «не забыть»;
- «запиши»;
- «добавь в план»;
- «потом надо будет»;
- «это задача на будущее»;
- «вернёмся к этому»;
- «это пункт плана»;
- «продолжи по TODO»;
- «обнови статус задачи».

Многошаговость — только один из сигналов, но не обязательное условие.

Если задача должна пережить текущую сессию, она должна попасть в `TODO.md`.

Если задача не должна пережить текущую сессию, её не надо превращать в пункт `TODO.md`.

## 4. Session todo и проектный TODO.md

Codex должен различать два уровня задач.

### 4.1. Session todo

Session todo — это короткий рабочий список шагов текущего запуска.

Он нужен, когда Codex прямо сейчас выполняет задачу:

- прочитать файлы;
- составить план;
- внести правку;
- проверить;
- отчитаться.

Session todo не обязан сохраняться в файл, если задача короткая и не должна жить после завершения текущей сессии.

### 4.2. Project tasks

Project tasks — это задачи проекта, которые должны переживать текущую сессию.

Они фиксируются в `TODO.md`, если:

- пользователь просит записать задачу;
- пользователь говорит, что задачу нужно сделать позже;
- задача является пунктом плана;
- задача должна быть продолжена в будущем;
- задача не закрывается в текущем запуске;
- задача требует проверки позже;
- появился блокер;
- есть риск потерять незавершённый шаг;
- нужно передать состояние в следующую сессию.

## 5. Канонические task-файлы

Канонический файл открытых задач:

`TODO.md`

Путь закрытой истории для нового проекта по умолчанию:

`archive/tasks/YYYY-MM.md`

Фактический путь архива является проектной настройкой в корневом `AGENTS.md`. Project skill читает его оттуда и не хранит собственную копию пути.

Архивный файл создаётся только при первом закрытии или миграции задачи за соответствующий месяц.

Не создавать другие task-файлы без явного запроса пользователя.

## 6. Когда вести TODO.md

Codex должен работать с `TODO.md`, если задача:

- явно должна быть записана;
- явно должна быть выполнена позже;
- является долговременным пунктом проектного плана;
- требует начать или продолжить уже отслеживаемую работу;
- требует изменить состояние существующей задачи;
- содержит блокер;
- требует проверки как отдельной будущей задачи;
- может занять больше одной сессии;
- должна быть понятна в будущей сессии.

Типовые ситуации:

- пользователь просит добавить задачу на будущее;
- пользователь просит не забыть задачу;
- пользователь просит составить проектный план, который нужно продолжать в будущих сессиях;
- пользователь просит выполнить конкретный пункт из плана;
- пользователь просит продолжить работу;
- нужно восстановить состояние project task для продолжения, закрытия, проверки proof или поиска дубля;
- пользователь сообщает, что часть существующей project task выполнена;
- Codex обнаруживает, что задача не может быть завершена сейчас;
- для существующей project task после правок нужно сохранить proof.

Простое справочное чтение не включает управление задачами. Список и
фильтрацию задач, проверку наличия записи, краткое объяснение текущей записи и
точечный поиск результата закрытой задачи можно выполнить напрямую, прочитав
только релевантные секции.

## 7. Когда НЕ вести TODO.md

Codex не должен трогать `TODO.md`, если задача:

- одноразовый короткий вопрос без будущего действия;
- обычное объяснение;
- перевод;
- простая справка;
- маленькая правка без продолжения;
- генерация одного текста без дальнейшего плана;
- быстрый ответ, который не создаёт долговременного состояния;
- обсуждение идеи без решения превратить её в задачу;
- черновой мозговой штурм без статуса и критерия готовности.

Не записывать в `TODO.md` каждую мысль из разговора. Записывать только то, что стало задачей.

## 7.1. Режимы включения TODO.md

Codex должен использовать три режима работы с `TODO.md`.

### 7.1.1. No TODO mode

Использовать для коротких, самодостаточных задач, которые не должны переживать текущую сессию.

В этом режиме Codex:

- не останавливает пользователя ради декомпозиции;
- не требует сначала записать задачу в `TODO.md`;
- не обновляет `TODO.md`;
- может держать короткий рабочий план внутри текущей сессии, если это помогает выполнить задачу.

Codex не должен говорить пользователю: “сначала давайте запишем это в TODO”, если задача выглядит короткой и не создаёт долговременного состояния.

### 7.1.2. Soft capture mode

Использовать для спонтанных задач, которые начались без project task, но по ходу работы приобрели долговременное состояние.

Codex должен мягко зафиксировать задачу в `TODO.md` в естественной точке, если выяснилось, что задача:

- не закрылась в текущем запуске;
- породила самостоятельный будущий шаг;
- требует проверки позже;
- стала долговременной частью проектного плана;
- получила блокер;
- должна быть продолжена в будущем.

Размер задачи, число шагов или количество затронутых файлов сами по себе не включают `Soft capture`. Решает необходимость сохранить состояние между сессиями.

Soft capture не должен мешать выполнению. Если работа уже идёт, Codex может обновить `TODO.md` перед финальным ответом или на ближайшем естественном checkpoint.

### 7.1.3. Tracked mode

Использовать, когда пользователь явно просит:

- записать задачу;
- не забыть;
- добавить на потом;
- добавить в план;
- продолжить по `TODO.md`;
- выполнить задачу из `TODO.md`;
- обновить статус;
- закрыть задачу.

В этом режиме Codex обязан работать через `TODO.md`:

1. сначала прочитать релевантную задачу;
2. обновить статус или `Current Action`;
3. потом выполнять работу;
4. после работы определить фактический результат;
5. если задача не завершена, обновить `Status`, `Current Action`, `Proof`, блокеры и `Next`;
6. если задача выполнена или отменена, выполнить closure-pass: записать её в архив и удалить из `TODO.md`;
7. перед финальным ответом выполнить один итоговый validation-pass для всех затронутых task-файлов.

Главное правило: `TODO.md` не должен быть турникетом перед каждой мелкой задачей. Он должен быть рабочей памятью проекта для того, что нельзя потерять.

## 8. Минимальные статусы

Использовать только эти статусы:

- `pending` — задача ожидает работы;
- `in_progress` — задача сейчас в работе;
- `blocked` — задача заблокирована;
- `done` — задача выполнена;
- `cancelled` — задача отменена.

Не использовать отдельные статусы для проверки или полной системной приёмки.

Если нужно что-то проверить, перепроверить или протестировать, это должно быть отдельной задачей в `TODO.md`, а не отдельным статусом основной задачи.

Пример разделения закрытой задачи и будущей проверки:

```md
<!-- archive/tasks/YYYY-MM.md -->

### Task 1

Status: done

#### Proof

- Implementation completed and validation passed.

<!-- TODO.md -->

##### Task 2

Status: pending

###### Goal

- Verify the completed implementation in a real scenario.

###### Next

- Run the relevant check or test.
```

Если работа частично выполнена, не вводить отдельный статус. Оставить задачу `in_progress` или `blocked` и явно написать, что уже сделано и что осталось.

Значения статусов не меняются после добавления архива. Меняется только место хранения закрытой записи: задача со статусом `done` или `cancelled` переносится в архив и удаляется из `TODO.md`.

## 9. Done, cancelled и proof

`done` означает: задача выполнена.

У задачи со статусом `done` желательно иметь минимальный proof, если задача меняла файлы, запускала проверки или влияла на проектное состояние.

`cancelled` означает: задача отменена. Для неё нужно кратко сохранить причину отмены; proof требуется только если он действительно полезен.

Proof может быть:

- изменённый файл;
- применённый патч;
- просмотренный diff;
- успешная команда проверки;
- read-back после записи;
- подтверждение пользователя;
- ссылка на конкретный артефакт.

Proof не может быть:

- обсуждением задачи;
- планом;
- намерением;
- обещанием Codex;
- самим наличием пункта в `TODO.md`;
- фразой “похоже, сделано”;
- отсутствием ошибки без проверки.

Если proof отсутствует, Codex не должен выдумывать его. При необходимости нужно создать отдельную задачу проверки.

После закрытия задачи её итоговый статус, результат или причина отмены и proof сохраняются в `archive/tasks/YYYY-MM.md`. Постоянно держать закрытую задачу в `TODO.md` нельзя.

## 10. Active, Later и архив

Каждая область внутри `TODO.md` может иметь две рабочие секции: `Active` и `Later`. Архив находится вне `TODO.md`.

### 10.1. Active

`Active` — задачи в работе или рядом с текущей работой.

Сюда попадают:

- текущие задачи;
- ближайшие задачи;
- задачи, которые пользователь просит выполнить сейчас;
- задачи, которые уже начаты;
- задачи, где есть блокер и нужен следующий шаг.

### 10.2. Later

`Later` — задачи на потом.

Сюда попадают:

- задачи, которые нужно не забыть;
- задачи, которые пользователь просит сделать позже;
- идеи, которые уже стали задачами;
- будущие улучшения с понятной целью;
- отложенные пункты плана.

`Later` не должен становиться свалкой идей. Если у пункта нет цели и следующего действия, он ещё не готов как задача.

### 10.3. Архив закрытых задач

Закрытые задачи не образуют постоянный раздел `Done` внутри `TODO.md`.

После получения статуса `done` или `cancelled` запись задачи переносится в месячный файл:

`archive/tasks/YYYY-MM.md`

Месяц определяется по дате закрытия. При миграции старых задач использовать `Updated`, если дата доступна.

Архив хранит краткую запись закрытия:

- название задачи;
- проект и область;
- финальный статус;
- дату создания, если она известна;
- дату закрытия;
- результат или причину отмены;
- минимальный proof;
- важные файлы или артефакты;
- follow-up, если он есть.

Архив не должен содержать полный диалог, пошаговый transcript, скрытые рассуждения или длинный пересказ выполнения.

### 10.4. Когда читать архив

Архив не нужно читать при каждой задаче.

Codex обращается к `archive/tasks/`, когда:

- пользователь спрашивает, что было сделано раньше;
- новая задача зависит от прежней работы;
- нужно найти результат или proof закрытой задачи;
- нужно проверить, не выполнялась ли похожая задача;
- нужно восстановить историю работы по проекту или области;
- при закрытии задачи нужно проверить возможный дубль архивной записи.

Сначала искать по названию задачи, проекту, области или ключевым словам. Читать только подходящие месячные файлы, а не весь архив.

Точечный поиск результата закрытой задачи является обычным справочным чтением.
Если архив нужен для продолжения или закрытия задачи, проверки proof,
восстановления project state или поиска дубля, использовать
`manage-project-tasks`. Само чтение `TODO.md` или архива не делает этот skill
владельцем другой работы.

## 11. Базовый формат задачи

Задача внутри `Active`:

```md
##### <task-name>

Status: pending
Created: YYYY-MM-DD
Updated: YYYY-MM-DD

###### Goal

- <what should become true>

###### Current Action

- <what is happening now, only when the task is in_progress>

###### Items

- [ ] pending: <item>
- [~] in_progress: <item>
- [!] blocked: <item> — reason: <reason>
- [x] done: <recently completed item, only until it is summarized in Progress>
- [-] cancelled: <item> — reason: <reason>

###### Done When

- <clear acceptance criterion>

###### Proof

- <minimal evidence available while the task remains open>

###### Next

- <next action>
```

Задача внутри `Later` может быть короче:

```md
##### <future-task-name>

Status: pending
Created: YYYY-MM-DD
Updated: YYYY-MM-DD

###### Goal

- <what should become true later>

###### Next

- <first useful action>
```

Закрытая задача хранится в `archive/tasks/YYYY-MM.md`:

```md
# Task Archive — YYYY-MM

## YYYY-MM-DD

### <task-name>

Status: done
Project: <project-name>
Area: <area-name>
Created: YYYY-MM-DD
Closed: YYYY-MM-DD

#### Result

- <what changed>

#### Proof

- <minimal proof, if available>

#### Files / Artifacts

- `<path-or-artifact>`

#### Follow-up

- <new task in TODO.md, or "None">
```

Для `cancelled` вместо `Result` использовать:

```md
#### Cancellation Reason

- <why the task was cancelled>
```

Пока задача открыта, `Proof` хранит доступные на текущий момент подтверждения. После закрытия итоговый proof переносится в архив вместе с задачей.

## 12. Полный обезличенный пример TODO.md

Этот пример только объясняет структуру и не используется как устанавливаемый шаблон. При установке брать точные форматы только из канонических блоков разделов 18.8-18.14 и не копировать демонстрационные значения как реальные проектные задачи.

```md
# TODO.md

## Project: Project 1

### Purpose

- Explain why Project 1 exists.

### Area: Area 1

#### Purpose

- Explain why Area 1 exists inside Project 1.

#### Active

##### Task 1

Status: in_progress
Created: YYYY-MM-DD
Updated: YYYY-MM-DD

###### Goal

- Describe what should become true after Task 1 is completed.

###### Current Action

- Describe the current meaningful step.

###### Items

- [~] in_progress: Work on the current meaningful step.
- [ ] pending: Do the next step after that.

###### Done When

- Define the acceptance condition.

###### Proof

- Add minimal proof available while the task remains open.

###### Next

- Continue with the pending step.

#### Later

##### Task 2

Status: pending
Created: YYYY-MM-DD
Updated: YYYY-MM-DD

###### Goal

- Describe a future task that should not be forgotten.

###### Next

- Start when this task becomes relevant.

```

Закрытая `Task 1` после closure-pass больше не находится в `TODO.md`. Её компактная запись находится в соответствующем `archive/tasks/YYYY-MM.md`.

## 13. Строгий формат и допустимая гибкость

Формат должен быть достаточно строгим, чтобы Codex мог стабильно обновлять задачи.

Для активной задачи обязательны:

- название задачи;
- `Status`;
- `Created`;
- `Updated`;
- `Goal`;
- `Items`;
- `Done When`;
- `Proof`;
- `Next`.

Для задачи со статусом `in_progress` обязателен блок:

- `Current Action`.

Для задач в `Later` допустим сокращённый формат:

- название задачи;
- `Status`;
- `Created`;
- `Updated`;
- `Goal`;
- `Next`.

Допускается мягкое расширение, если задача не укладывается в базовую форму.

Можно добавлять:

- `Progress`;
- `Context`;
- `Constraints`;
- `Decisions`;
- `Risks`;
- `Open Questions`;
- `Related Files`.

Нельзя превращать `TODO.md` в длинный журнал событий.

Правила компактности:

- `Items` хранит текущую и оставшуюся работу, а не полный список завершённых действий;
- не накапливать длинный список выполненных подпунктов;
- выполненные подпункты сворачиваются в `Progress` максимум на 1–3 пункта;
- после сворачивания удалять соответствующие `[x] done` из `Items`;
- длинный план выносится в отдельный документ только при реальной необходимости, а в задаче остаётся ссылка;
- после закрытия результат и proof переходят в архив;
- закрытые задачи не остаются в `Active` или `Later`.

## 13.1. Сложные и неопределённые задачи

Если полный маршрут нельзя честно распланировать до конца, важные вводные
появятся только по ходу, несколько жизнеспособных направлений остаются
открытыми или текущий план мог устареть, `TODO.md` всё равно не должен
превращаться в карту мышления, журнал сигналов или длинный план.

В таких случаях Codex может использовать отдельный метод работы с
неопределённостью, если он установлен в проекте, но `TODO.md` хранит только
компактное task-состояние. Сложность, стратегичность, риск или многошаговость
сами по себе не включают этот метод, если маршрут и критерий готовности уже
ясны.

Практическое соответствие:

- `Goal` хранит целевой контур задачи: что должно стать правдой и что нельзя потерять.
- `Current Action` хранит текущую ближайшую итерацию или рабочий шаг.
- `Items` хранит текущую и оставшуюся работу, а не полную историю итераций.
- `Progress` может кратко фиксировать последний значимый сигнал или уже закрытый участок, максимум 1–3 пункта.
- `Constraints`, `Risks`, `Open Questions` и `Decisions` можно добавлять только если они реально помогают продолжить задачу в будущей сессии.
- `Next` хранит следующий полезный шаг после текущей итерации.
- `Proof` хранит минимальное подтверждение текущего состояния или результата, но не рассуждения.

Не записывать в `TODO.md`:

- полный целевой контур, если он уже живёт в отдельном рабочем документе;
- подробную карту решений;
- журнал всех сигналов;
- цепочку рассуждений;
- сырой Zoom out;
- все гипотезы подряд;
- промежуточный шум;
- длинный план дальнего горизонта.

Если сложная задача требует отдельной карты решений, журнала сигналов, списка гипотез или рабочего плана, Codex может предложить отдельный рабочий документ. Создавать его можно только при реальной необходимости или по явному запросу пользователя.

Короткое правило:

`Метод работы с неопределённостью управляет движением. TODO.md фиксирует только то, что нужно не потерять.`

## 14. Правило начала работы над задачей

Это правило действует для `Tracked mode`: когда Codex приступает к задаче, уже находящейся в `TODO.md`, или пользователь явно просит работать по `TODO.md`.

В этом режиме Codex должен сначала обновить `TODO.md`, а потом выполнять реализацию.

Порядок:

1. Прочитать релевантную задачу в `TODO.md`.
2. Перевести задачу или пункт в `in_progress`.
3. Записать `Current Action`: что именно Codex делает сейчас.
4. Выполнить работу.
5. Если работа не завершена, обновить задачу и оставить статус `in_progress` или `blocked`.
6. Если задача завершена или отменена, выполнить closure-pass: записать `done` или `cancelled` в архив и удалить задачу из `TODO.md`.
7. Если нужна отдельная проверка, создать отдельную задачу проверки.
8. Перед финальным ответом выполнить один validation-pass для всех изменений.

Для `No TODO mode` это правило не применяется.

Для `Soft capture mode` Codex может обновить `TODO.md` не перед стартом, а в ближайшей естественной точке: когда стало ясно, что задача должна сохраниться между сессиями, или перед финальным ответом.

Codex не должен задним числом делать вид, что задача велась как tracked-задача, если она изначально не была такой. В этом случае нужно честно указать, что задача была зафиксирована через soft capture.

## 15. Блокеры и зависимости

Блокеры использовать аккуратно.

Не строить систему зависимостей по умолчанию.

Статус `blocked` использовать только когда задача реально не может двигаться без:

- ответа пользователя;
- доступа;
- файла;
- решения;
- внешнего результата.

Формат:

```md
- [!] blocked: <item> — reason: <short reason>
```

Не вводить сложные dependency-id, lanes, owners или тяжёлую проектную структуру без явного запроса пользователя.

## 16. Архив закрытия и отдельные рабочие файлы

Сырой отдельный лог действий по умолчанию не нужен.

`archive/tasks/YYYY-MM.md` является частью системы, но создаётся лениво: только при первом закрытии или миграции задачи за соответствующий месяц.

Архив закрытия не равен `progress.md` и не является transcript. Он хранит только компактные записи закрытых или отменённых задач.

Отдельный файл состояния, прогресса или передачи нужен только при явной необходимости:

- задача очень длинная;
- много независимых потоков;
- требуется отдельная история решений;
- нужно передавать работу другому исполнителю;
- длинный процесс требует самостоятельного текущего состояния вне задач.

Если такой необходимости нет, не множить сущности.

Не создавать автоматически:

- `state.md`;
- `progress.md`;
- `handoff.md`;
- `CHANGELOG.md`;
- ADR;
- отдельный технический лог.

### 16.1. Closure-pass

Closure-pass выполняется, когда задача получает статус `done` или `cancelled`.

Порядок:

1. Собрать компактную запись закрытия из задачи.
2. Прочитать действующий путь архива из корневого `AGENTS.md` и определить месячный файл по дате закрытия.
3. Проверить, нет ли уже записи с теми же `Project`, `Area`, названием задачи и `Created`.
   Если `Created` отсутствует, использовать исходный `Updated` или дату закрытия как часть ключа.
4. Если запись уже есть, обновить её вместо добавления дубля.
5. Записать финальный статус, результат или причину отмены, proof, важные файлы и follow-up.
6. Прочитать архивную запись обратно и проверить её.
7. Только после успешной проверки удалить закрытую задачу из `TODO.md`.
8. Если осталась самостоятельная будущая работа, создать новую задачу со статусом `pending`.

Запись в архив и удаление из `TODO.md` считаются одной логической операцией. Если архив не удалось записать, закрытую задачу нельзя удалять из `TODO.md`.

### 16.2. Cleanup-pass

Cleanup-pass используется для существующего разросшегося `TODO.md`.

Codex должен:

1. Найти задачи со статусом `done` или `cancelled`.
2. Перенести их в месячные архивы без потери результата, причины отмены и proof.
3. Не создавать дубли уже существующих архивных записей.
4. Прочитать и проверить созданные или обновлённые архивные записи.
5. Удалить из `TODO.md` только успешно проверенные закрытые задачи.
6. Удалить пустые `Done` и пустые области только если в них не осталось открытой работы или нужного `Purpose`.
7. Не переписывать и не нормализовать нерелевантные открытые задачи без необходимости.

## 17. Правила обновления TODO.md

Перед началом работы Codex должен:

1. Определить режим: `No TODO`, `Soft capture` или `Tracked`.
2. Если выбран `No TODO mode`, не читать и не обновлять `TODO.md` без необходимости.
3. Если задача относится к `TODO.md`, прочитать релевантную задачу и окружающий `Project`, `Area`, `Active` или `Later`, а не весь файл без необходимости.
4. Найти релевантный `Project` и `Area`.
5. Если релевантной области нет, создать её только при реальной необходимости.
6. Найти релевантную активную задачу или задачу в `Later`.
7. Если релевантной задачи нет, создать новую только при реальной необходимости.
8. Не трогать нерелевантные проекты, области и задачи.

Во время работы Codex должен:

1. Обновлять только релевантную секцию.
2. Не переписывать весь `TODO.md` без необходимости.
3. Не удалять незавершённую задачу без понятной причины.
4. Обычное обновление задачи, перенос закрытой задачи в архив и удаление явного дубля не требуют отдельного разрешения пользователя.
5. Если удаление меняет согласованный план, уничтожает самостоятельную незавершённую задачу или может привести к потере информации, сначала кратко объяснить изменение и запросить одно подтверждение.
6. Не менять статус на `done` без фактического выполнения.
7. Не выдумывать proof.
8. Если нужна проверка, создавать отдельную задачу проверки.
9. Фиксировать блокеры коротко и конкретно.
10. Фиксировать следующий шаг.
11. Не выполнять read-back после каждого промежуточного обновления.

Перед финальным ответом Codex должен:

1. Сделать reconciliation затронутых задач.
2. Обновить `Updated`.
3. Для `done` или `cancelled` выполнить closure-pass.
4. Записать все связанные изменения одним пакетом, насколько это возможно.
5. Выполнить один итоговый validation-pass:
   - прочитать затронутые секции `TODO.md`;
   - прочитать затронутый месячный архив, если он изменялся;
   - проверить отсутствие дубля архивной записи;
   - проверить, что закрытая задача больше не находится в `TODO.md`;
   - проверить, что открытые задачи и их статусы не потеряны.
6. Не повторять тот же read-back, если validation-pass прошёл.
7. Кратко сообщить пользователю, что изменено в task-файлах.

## 18. Установка в проект

Когда пользователь передаёт этот документ и просит установить систему, Codex должен выполнить setup-pass.

Разделы 1-17 объясняют модель и ожидаемое поведение системы. Они не являются отдельным устанавливаемым runtime-владельцем. Для записи файлов использовать только канонические блоки разделов 18.8-18.14. При расхождении пояснительного примера с каноническим артефактом приоритет имеет канонический артефакт; после установки рабочими владельцами становятся корневой `AGENTS.md`, `manage-project-tasks`, `TODO.md` и действующий архив.

### 18.1. Граница установки

Установка изменяет только проектные task-артефакты:

- корневой проектный `AGENTS.md`;
- проектный skill `.agents/skills/manage-project-tasks/`;
- корневой `TODO.md` с точным регистром имени;
- действующий каталог task-архива, только если при установке реально архивируются закрытые задачи.

Каноническое имя файла открытых задач:

`TODO.md`

Для нового проекта канонический путь архива:

`archive/tasks/YYYY-MM.md`

Если в существующем проекте уже используется другой явно действующий путь архива, сохранить его и подставить этот конкретный путь во все устанавливаемые правила. Не создавать второй архив только ради соответствия пути по умолчанию.

Если найден файл `TODO.MD`, `todo.md` или другой вариант регистра того же имени, считать его кандидатом существующей task-системы. До записи определить, является ли файловая система чувствительной к регистру. Без потери содержимого привести имя к точному `TODO.md`; на нечувствительной к регистру файловой системе использовать безопасное промежуточное имя. Не оставлять одновременно два файла, различающихся только регистром.

Не изменять глобальный `~/.codex/AGENTS.md`, `AGENTS.override.md` или вложенные `AGENTS.md` без явного запроса пользователя.

Если корень проекта неясен, сначала определить его по рабочей папке, Git root или явному указанию пользователя. Не угадывать другой проект.

Git не является требованием установки. Если Git отсутствует, использовать рабочую папку или явно указанный путь и не выполнять Git-команды.

### 18.2. Что прочитать

Codex должен прочитать:

1. Этот документ.
2. Корневой проектный `AGENTS.md`, если он есть.
3. Корневой `TODO.md` или кандидат с другим регистром, если он есть.
4. Список существующих файлов в действующем task-архиве, если каталог есть.
5. Конкретный месячный архив только если он нужен для миграции или проверки дубля.
6. Существующий `.agents/skills/manage-project-tasks/`, если он есть.
7. Активный глобальный `skill-creator` полностью, включая правила `openai.yaml` и `quick_validate.py`.
8. README или другие проектные инструкции только если без них нельзя определить корень, стиль или безопасное место вставки.

Не читать весь проект по умолчанию.

### 18.3. Один inspect-pass перед записью

Перед изменениями Codex должен одним проходом определить:

- найден ли корневой `AGENTS.md`;
- есть ли в нём существующая task/TODO-политика;
- существует ли `.agents/skills/manage-project-tasks/`;
- является ли существующий одноимённый skill предыдущей совместимой установкой или отдельным пользовательским skill;
- найден ли `TODO.md` или вариант имени с другим регистром;
- какой точный путь будет использоваться как `<project-root>/TODO.md`;
- какой конкретный путь архива уже является действующим или будет создан для нового проекта;
- какие открытые и закрытые статусы в нём присутствуют;
- есть ли постоянные разделы `Done`;
- существуют ли task-архивы;
- существуют ли альтернативные task-источники: `TASKS.md`, `tasks.md`, backlog, `logs/tasks/` или внешний трекер;
- можно ли безопасно сохранить все текущие задачи и proof;
- какие файлы будут созданы или изменены.

Установка должна различать четыре состояния:

1. Новая установка без task-системы.
2. Старая установка этой системы, где полная процедура находится в `AGENTS.md`, а project skill отсутствует.
3. Совместимая установка с `manage-project-tasks`, которую можно обновить на месте.
4. Другая, адаптированная или конфликтующая task-система.

Перед записью зафиксировать компактный снимок исходного task-состояния:

- точное имя и путь task-файла;
- названия открытых задач;
- `Project` и `Area`;
- статусы;
- `Created` и `Updated`, если они есть;
- `Proof`, если он есть;
- количество открытых задач;
- найденные закрытые задачи, если требуется миграция.

Снимок нужен только для сравнения до/после в текущем проходе. Не создавать для него постоянный файл состояния.

Если проект новый и task-системы ещё нет, Codex может выполнить установку без дополнительного вопроса.

Если task-система уже существует и для обновления нужны заметные структурные изменения, Codex должен один раз:

1. Кратко описать, что найдено.
2. Предложить безопасные варианты обновления.
3. Объяснить, какие файлы и данные затронет каждый вариант.
4. Рекомендовать один вариант.
5. Дождаться выбора или подтверждения пользователя.

После выбора выполнить согласованный вариант целиком. Не запрашивать отдельное разрешение на каждую задачу, поле или обычное обновление статуса.

Остановиться и спросить пользователя только если:

- корень проекта неясен;
- две существующие политики прямо противоречат друг другу и нельзя безопасно выбрать одну;
- структура `TODO.md` не позволяет надёжно отделить задачи;
- проект использует другой явный источник истины задач и переход на `TODO.md` не был согласован;
- миграция может потерять или перепутать данные.

### 18.4. Как распознать существующую систему

Codex должен определять систему по содержанию файлов, а не по служебному номеру версии.

Старая установка этой системы обычно содержит:

- корневой `TODO.md`;
- подробную самодостаточную task-политику и точные шаблоны внутри корневого `AGENTS.md`;
- отсутствие project skill `manage-project-tasks`;
- уровни `Project` и `Area`;
- секции `Active`, `Later` и постоянный `Done`;
- статусы `pending`, `in_progress`, `blocked`, `done`, `cancelled`;
- поля `Current Action`, `Items`, `Done When`, `Proof` и `Next`;
- правило read-back после каждого изменения.

Другая или адаптированная task-система может содержать:

- несколько TODO-файлов с разными зонами ответственности;
- локальные TODO для клиентов, подпроектов или процессов;
- другой основной источник истины;
- собственную маршрутизацию задач;
- специализированный backlog или внешний трекер.

Такую систему нельзя автоматически считать старой установкой и заменять целиком.

При обновлении старой установки этой системы предложить пользователю варианты, например:

1. Установить новую архитектуру `AGENTS.md` + `manage-project-tasks`, а перенос закрытых задач выполнить позже.
2. Установить новую архитектуру и сразу перенести закрытые задачи в месячные архивы.
3. Оставить текущую систему без изменений.

Для адаптированной или другой task-системы сначала отдельно предложить, как встроить архив закрытых задач без разрушения её зон ответственности.

Согласованный upgrade-pass должен:

1. Обновить одну существующую TODO-политику в `AGENTS.md`, а не добавлять вторую.
2. Установить или обновить один канонический project skill `manage-project-tasks`.
3. Перенести точные шаблоны из корневой политики в канонический skill reference, не оставляя две активные подробные версии.
4. Перенести закрытые задачи из постоянного `Done`, только если пользователь выбрал этот вариант.
5. Распознать старый путь вроде `logs/tasks/YYYY-MM.md`, но не удалять и не переименовывать такой файл автоматически.
6. Сохранить открытые задачи, статусы, даты, proof и project/area-привязку.
7. Не оставлять одновременно две активные политики, если пользователь согласовал замену старой политики.
8. Кратко сообщить, что было изменено и какие прежние отдельные файлы остались на месте.

# Manage Project Tasks

Own the runtime procedure for durable project tasks. Keep open work in
`TODO.md`, closed work in the task archive path declared by root `AGENTS.md`,
and detailed task history out of chat memory.

Root `AGENTS.md` owns the always-loaded trigger, allowed statuses, and critical
safety invariants. This skill owns how to carry them out.

## Choose The Mode

Choose exactly one mode for the current work:

- `No TODO`: use for short self-contained work that should not survive the
  session. Do not read or modify task files.
- `Soft capture`: use when spontaneous work grows, remains unfinished, becomes
  blocked, creates independent follow-up, or otherwise must survive the
  session. Capture it at the nearest natural checkpoint; do not pretend it was
  tracked from the start.
- `Tracked`: use when the user explicitly asks to add, remember, continue,
  update, block, close, cancel, archive, or reconstruct durable task state, or
  when work starts from an existing `TODO.md` task.

The number of implementation steps does not determine the mode. Durability
does.

## Read Before Mutation

Before changing `TODO.md` or the task archive:

1. Read the affected task and its surrounding `Project`, `Area`, `Active`, or
   `Later` context.
2. Read [references/task-templates.md](references/task-templates.md) when
   creating a task, repairing task structure, closing or cancelling a task, or
   writing an archive record.
3. Read
   [references/system-maintenance.md](references/system-maintenance.md) only
   when changing the task system, migrating task data, cleaning up several
   tasks, or handling an uncertain structural case.
4. Read only matching archive records or monthly files. Do not scan the entire
   archive during ordinary work.

Do not rely on conversation memory, a compacted summary, or an earlier reading
for exact templates, archive shape, or closure order. The task operation
itself, not awareness of compaction, triggers the required read.

## Create Or Capture A Task

1. Confirm that the work must survive the session.
2. Find the relevant existing `Project` and `Area`; create either only when
   genuinely necessary.
3. Check the nearby open tasks for an obvious duplicate.
4. Put current, near-current, or blocked work in `Active`; put future work in
   `Later`.
5. Use only `pending`, `in_progress`, `blocked`, `done`, or `cancelled`.
6. Use the applicable canonical template from `task-templates.md`.
7. Write only durable state. Do not preserve chat narration or speculative
   detail.

An explicitly requested future task normally starts as `pending` in `Later`.
Soft-captured work may start in `Active` when work is already underway.

## Start Or Continue Tracked Work

Before implementation:

1. Read the relevant task.
2. Move it to `Active` when needed.
3. Set `Status: in_progress`.
4. Update `Updated`.
5. Write one concrete `Current Action`.
6. Reconcile `Items` so one current item is marked `[~] in_progress` and
   remaining work is honest.

During and after work:

- update only the affected task and surrounding structure;
- keep `Items` focused on current and remaining work;
- compact completed items into at most one to three `Progress` bullets, then
  remove the corresponding `[x]` items;
- put only available evidence in `Proof`;
- keep `Next` concrete;
- leave the task `in_progress` or `blocked` when work remains.

Multiple tasks may be `in_progress`, but do not mark unrelated work active.

## Record A Blocker

Use `blocked` only when the task cannot proceed without a concrete input,
decision, file, access grant, or external result.

Record:

- the blocked item;
- the specific reason;
- completed progress that must not be lost;
- the event or input that will unblock the task;
- the next safe action, if one exists.

Do not invent dependency graphs, owners, lanes, or new statuses.

## Close Or Cancel

Treat closure as one logical operation:

1. Confirm the factual final status and available evidence.
2. Read the closure and archive templates in `task-templates.md`.
3. Read the active archive pattern from root `AGENTS.md` and determine the
   monthly archive path from the closure month.
4. Check for an existing record using `Project`, `Area`, task title, and
   `Created`. If `Created` is missing, use the source `Updated` or closure date
   as the stable migration date.
5. Append or update one compact archive record.
6. Read back and verify the archive record.
7. Only after successful archive verification, remove the closed task from
   `TODO.md`.
8. Create a separate `pending` task when independent follow-up remains.

If archive writing or verification fails, keep the task in `TODO.md`.

`done` means the work was performed. `cancelled` means it was intentionally
abandoned. Plans, discussion, promises, task existence, or "probably done" are
not proof.

Routine task updates and normal closure do not need separate permission. Ask
for explicit confirmation before deleting or merging an independent unfinished
task, changing an agreed plan, or taking an action that may lose information.

## Reconstruct Task State Or History

Simple read-only listing, filtering, existence checks, brief explanation of a
current entry, and targeted lookup of a closed result do not require this
skill. Read only the relevant task sections directly when no durable state
must be changed or reconstructed.

Use this skill when current or historical task state must be reconstructed to
resume work, close or cancel a task, verify proof, detect duplicates, rebuild a
project's task state, or deduplicate closure. Search first by title, project,
area, or keywords, then read only the relevant `TODO.md` sections and matching
monthly archive files.

## Integrate With Nearest Clarity

`nearest-clarity` controls movement through uncertainty. This skill stores only
durable task state.

Using `TODO.md` as an information source does not make this skill the parent
workflow. When the task is to choose a direction under genuine uncertainty,
`nearest-clarity` owns the work. Invoke this skill only if the resulting
durable task state must be persisted or reconstructed.

For work that must survive the session:

- `Goal` may hold the target contour;
- `Current Action` holds the current iteration or concrete working step;
- `Progress` holds compact meaningful results;
- `Next` holds the next useful action;
- add `Constraints`, `Risks`, `Open Questions`, or `Decisions` only when
  necessary for continuation.

Do not turn `TODO.md` into a reasoning transcript, signal journal, hypothesis
dump, detailed decision map, or distant plan.

## Validate Task Changes

Batch related writes, then perform one final validation pass:

1. Read back the changed task sections.
2. Read back each changed monthly archive.
3. Confirm every removed closed task exists in the archive exactly once.
4. Confirm open tasks and their statuses remain in `TODO.md`.
5. Confirm required fields and allowed statuses.
6. Check UTF-8 readability after Cyrillic writes.
7. Report briefly what changed and what was left untouched.

Do not repeat the same read-back after it passes.

## Boundaries

- Do not keep a permanent `Done` section in `TODO.md`.
- Do not create an empty archive in advance.
- Do not invent proof or claim validation that did not run.
- Do not rewrite unrelated open tasks for formatting consistency.
- Do not create competing task skills, task protocols, progress logs,
  handoff files, state files, changelogs, ADRs, dependency graphs, or extra
  project-management layers without a demonstrated need or explicit request.
- Do not change this task system as part of an ordinary task update.

## References

- [references/task-templates.md](references/task-templates.md): read for exact
  open-task, future-task, cancellation, and archive formats.
- [references/system-maintenance.md](references/system-maintenance.md): read
  for migration, cleanup, task-system changes, and isolated behavior tests.

Use these exact structures when creating, repairing, closing, or cancelling

## Open Task

## Project: <project-name>

### Purpose

- <why this project exists>

#### Purpose

- <why this area exists>

#### Active

##### <task-name>

Status: pending
Created: YYYY-MM-DD

###### Done When

- <acceptance condition>

###### Proof

- <minimal evidence available while the task remains open>

###### Next

- <next action>

For a full task in `Active`, required fields are `Status`, `Created`, `Updated`,

Existing project-specific routing may extend this structure. Preserve useful
extensions during routine updates.

## Future Task

```md
#### Later

##### <future-task-name>

Status: pending
Created: YYYY-MM-DD
Updated: YYYY-MM-DD

###### Goal

- <what should become true later>

###### Next

- <first useful action>
```

Expand the task to the full open-task structure when work begins or when the
extra fields become necessary.

## Archive File

```md
# Task Archive — YYYY-MM

## YYYY-MM-DD

### <task-name>

Status: done
Project: <project-name>
Area: <area-name>
Created: YYYY-MM-DD
Closed: YYYY-MM-DD

#### Result

- <what changed>

#### Proof

- <minimal evidence>

#### Files / Artifacts

- `<path-or-artifact>`

#### Follow-up

- <new TODO task, or "None">
```

Use the archive format with `Status: cancelled` and replace `Result` with:

```md

- <why the task was cancelled>
```

Proof is optional for cancellation unless it materially explains the final
state.

## Field Rules

- `Goal`: the state that should become true.
- `Current Action`: one concrete action being performed now.
- `Items`: current and remaining work, not a historical checklist.
- `Progress`: at most one to three compact completed results.
- `Done When`: acceptance condition.
- `Proof`: evidence already available while the task is open.
- `Next`: the next useful action.
- `Constraints`, `Risks`, `Open Questions`, and `Decisions`: add only when
  future continuation genuinely needs them.

Use only these statuses:

- `pending`
- `in_progress`
- `blocked`
- `done`
- `cancelled`

Knowledge labels such as fact, observation, hypothesis, decision, risk, noise,
or verified result are not task statuses.

# Task System Maintenance

Read this file only when migrating task data, changing task-system rules,
cleaning up several tasks, repairing structural drift, or testing the task

## Preserve Before Changing

Before a structural change:

1. Read root `AGENTS.md`, this skill, `TODO.md`, and only relevant archive

2. Record the current open task titles, statuses, dates, project and area
   placement, available proof, and active archive path.
3. Identify the current owner of every rule being changed.
4. Distinguish task data from formatting or protocol changes.
5. Do not alter unrelated project instructions or domain files.

For a grown or inconsistent `TODO.md`:

1. Find closed tasks still present in `TODO.md`.
2. Archive each through the normal closure procedure.
3. Avoid duplicate archive records.
4. Preserve all open tasks, statuses, dates, project and area placement, and
   available proof.
5. Remove empty `Done` sections and empty containers only when they hold no
   open work or useful `Purpose`.
6. Do not normalize unrelated open tasks merely for visual consistency.

## Changing The Task System

Keep exactly one detailed runtime owner: this skill.

When changing behavior:

1. Update the owning section here or in `SKILL.md`.
2. Keep root `AGENTS.md` limited to always-needed triggers and invariants.
3. Search active files for stale duplicates or conflicting rules.
4. Update project navigation when ownership or paths change.
5. Treat installation or portable source artifacts as separate owners; update
   them only when they are explicitly in scope.
6. Do not create a parallel protocol or second task skill.

## Structural Validation

After migration or cleanup, confirm:

- exactly one canonical `TODO.md` exists;
- only allowed statuses remain;
- open tasks, dates, placements, and available proof were preserved;
- closed tasks removed from `TODO.md` exist exactly once in the archive;
- no empty archive was created;
- root instructions point to this skill;
- references resolve;
- unrelated project instructions and task data remain intact;
- UTF-8 and Markdown are readable.

## Isolated Behavior Test

Use temporary copies, never live task files.

Minimum scenarios:

1. A short self-contained request leaves temporary task files unchanged.
2. "Add this for later" creates a `pending` task in `Later`.
3. Starting tracked work moves the task to `Active`, sets `in_progress`, and
   adds `Current Action`.
4. A real blocker sets `blocked`, preserves progress, and names the unblock

5. Closure writes and verifies the archive before removing the task from
   temporary `TODO.md`.
6. Simulated archive-write failure leaves the task in temporary `TODO.md`.
7. Repeated closure does not create a duplicate archive record.
8. Unrelated open tasks remain unchanged.

Report only the demonstrated level:

- `implemented`
- `structurally validated`
- `behavior smoke-tested`
- `live-tested`

Do not infer a higher level from a lower one.

4. Проверить существующий `.agents/skills/manage-project-tasks/`.
5. Классифицировать установку как новую, старую AGENTS-only, совместимую или конфликтующую.
6. Зафиксировать временный снимок открытых задач и согласовать заметную миграцию, если она нужна.
7. Если skill отсутствует, запустить:

```text
python "<skill-creator-dir>/scripts/init_skill.py" manage-project-tasks --path "<project-root>/.agents/skills" --resources references --interface display_name="Manage Project Tasks" --interface short_description="Ведение TODO и архива проектных задач" --interface 'default_prompt=Use $manage-project-tasks to safely update the current project task state.'
```

8. Если обнаружена совместимая установка, обновить её на месте без повторной инициализации.
9. Если одноимённый skill имеет другое назначение или неизвестные пользовательские расширения, остановиться до перезаписи и запросить решение.
10. Записать точные канонические `SKILL.md`, `agents/openai.yaml` и два reference-файла.
11. Заменить `{{TASK_ARCHIVE_PATTERN}}` в каноническом корневом разделе и встроить ровно один раздел `Project Task Tracking` в `AGENTS.md`.
12. Создать минимальный `TODO.md` с разрешённым путём архива либо совместимо сохранить существующий файл.
13. Выполнить только согласованную миграцию закрытых задач.
14. Не создавать пустой архив и не удалять самостоятельные прежние task-файлы.
15. Запустить:

```text
python "<skill-creator-dir>/scripts/quick_validate.py" "<project-root>/.agents/skills/manage-project-tasks"
```

16. Сравнить четыре установленных skill-файла с каноническими блоками этого документа.
17. Проверить корневой контракт, разрешённый путь архива и сохранность task-состояния.
18. Записать Markdown и YAML в UTF-8 и прочитать изменённые кириллические файлы обратно.

## 19. Проверка после установки

После setup-batch выполнить один согласованный validation-pass, но не смешивать разные уровни доказательств.

### 19.1. Проверка project skill

Проверить:

1. Существует ровно один `.agents/skills/manage-project-tasks/`.
2. В нём есть только канонические `SKILL.md`, `agents/openai.yaml`, два reference-файла и обязательные технические файлы среды, если она их создаёт.
3. `quick_validate.py` завершился успешно.
4. `SKILL.md`, `openai.yaml` и references совпадают с каноническими блоками установочника.
5. Все относительные ссылки из `SKILL.md` разрешаются.
6. `openai.yaml` содержит `$manage-project-tasks` и `allow_implicit_invocation: true`.
7. Frontmatter описывает запросы управления долговременными задачами и не содержит конкретный путь архива одного проекта.

### 19.2. Проверка установки

Прочитать изменённый раздел `AGENTS.md` и проверить:

- существует ровно один активный раздел task tracking;
- раздел разрешает простые справочные чтения напрямую и требует
  `manage-project-tasks` для mutations, task lifecycle и восстановления
  долговременного состояния;
- перечислены только статусы `pending`, `in_progress`, `blocked`, `done`, `cancelled`;
- указан фактический действующий путь архива;
- в установленных файлах не осталось `{{TASK_ARCHIVE_PATTERN}}`;
- в корневом разделе нет точных task-шаблонов, полного closure-pass, migration-pass или копии `SKILL.md`;
- корневой task-файл имеет точное имя `TODO.md`;
- архив не создан пустым без необходимости;
- прежние самостоятельные task-файлы и архивы не удалены;
- нерелевантные проектные инструкции не потеряны.

### 19.3. Проверка сохранности task-состояния

Сравнить снимок до и после:

- все открытые задачи сохранены;
- их статусы, даты, project/area-размещение и доступный proof сохранены, кроме явно согласованных изменений;
- `TODO.md` не использует постоянный `Done` для успешно перенесённых задач;
- каждая удалённая закрытая задача присутствует в архиве ровно один раз;
- ошибка записи или проверки архива не приводит к удалению задачи из `TODO.md`;
- пользовательские задачи не переписаны только ради форматирования.

Не выполнять повторные read-back или полный повторный аудит, если соответствующий слой проверки прошёл. После исправления повторять только затронутый слой.

### 19.4. Безопасный behavior smoke-test

После структурной проверки выполнить небольшой тест поведения, если среда позволяет создать и удалить временные файлы без влияния на проект.

Тест выполнять только на изолированной временной копии:

- не использовать настоящий `TODO.md`;
- не использовать настоящий task-архив;
- не менять реальные пользовательские задачи;
- не запускать рабочие маршруты проекта;
- удалить временную копию после успешной проверки.

Минимальные сценарии:

1. Короткий самодостаточный вопрос классифицируется как `No TODO`; временный `TODO.md` не меняется.
2. Запрос «добавь на потом» создаёт задачу `pending` в `Later`.
3. Начало tracked-задачи переносит её в `Active`, ставит `in_progress` и добавляет `Current Action`.
4. Реальный блокер ставит `blocked`, сохраняет сделанное и указывает конкретное условие разблокировки.
5. Закрытие сначала создаёт и проверяет архивную запись, затем удаляет задачу из временного `TODO.md`.
6. Имитированная ошибка записи архива не удаляет закрываемую задачу из временного `TODO.md`.
7. Повторный проход не создаёт дубль архивной записи.
8. После операций остальные открытые задачи, статусы, даты и proof остаются без изменений.

Smoke-test считается пройденным только если фактические временные файлы подтверждают все сценарии. Пересказ правил или утверждение агента не является доказательством.

Если среда не позволяет выполнить изолированный тест, не считать это ошибкой установки, но явно оставить статус `behavior smoke-test not run`.

### 19.5. Проверка маршрутизации

Проверка description включает как минимум:

Должны включать skill:

- «добавь это на потом»;
- «продолжи задачу из TODO»;
- «поставь задачу в blocked»;
- «закрой и архивируй задачу»;
- «восстанови состояние задачи, чтобы продолжить её»;
- «проверь proof перед закрытием задачи»;
- «проверь, не будет ли закрытие дублем»;
- спонтанная работа, которую нужно сохранить для следующей сессии.

Не должны включать skill сами по себе:

- «покажи текущие задачи»;
- «есть ли задача про X»;
- «объясни кратко эту запись»;
- «найди результат закрытой задачи X»;
- короткий самодостаточный вопрос;
- временный план действий текущей сессии;
- обычное сообщение о ходе работы без долговременного состояния;
- обсуждение идеи без решения сделать её задачей;
- использование `TODO.md` только как источника информации для другого
  процесса.

Статический разбор `description` подтверждает только покрытие формулировок. Фактический выбор skill проверять отдельно в свежем контексте. Не заявлять fresh-context routing tested, если такого запуска не было.

### 19.6. Уровни подтверждения

Использовать только подтверждённый текущими доказательствами уровень:

- `installed` — нужные проектные файлы записаны;
- `skill validated` — `quick_validate.py`, ссылки и канонические skill-файлы проверены;
- `structurally validated` — итоговые файлы прочитаны и проверены по инвариантам;
- `behavior smoke-tested` — изолированные временные сценарии фактически пройдены;
- `fresh-context routing tested` — выбор skill фактически проверен в независимом свежем контексте;
- `live-tested` — система прошла реальный жизненный цикл на настоящей пользовательской задаче.

Нельзя выводить более высокий уровень из более низкого.

Нормальная формула после полной установки и изолированного теста:

`installed + skill validated + structurally validated + behavior smoke-tested + fresh-context routing not tested + not live-tested`

Если smoke-test не запускался:

`installed + skill validated + structurally validated + behavior smoke-test not run + fresh-context routing not tested + not live-tested`

Не заявлять `live-tested` без отдельной реальной задачи, прошедшей полный жизненный цикл.

## 20. Работа после установки

После установки Codex использует корневой контракт для включения системы и
загружает `manage-project-tasks` для mutations, жизненного цикла долговременных
задач и восстановления состояния для продолжения, закрытия, проверки proof или
поиска дубля. Для простого справочного чтения skill не загружается: Codex
читает только релевантные секции напрямую.

Codex обновляет task-файлы, когда:

- начинается tracked-задача;
- меняется её фактический статус;
- появляется блокер;
- меняется следующий шаг;
- появляется proof;
- пользователь добавляет будущую задачу;
- soft capture превращает спонтанную работу в долговременную;
- задача закрывается или отменяется.

Не обновлять task-файлы при каждом маленьком ответе. Использовать осмысленные checkpoints и итоговый reconciliation.

## 21. Типовой жизненный цикл задачи

1. `pending` — задача добавлена.
2. `in_progress` — работа началась, записан `Current Action`.
3. `blocked` — задача не может двигаться без конкретного входа.
4. `done` — задача выполнена.
5. `cancelled` — задача отменена.
6. Closure-pass сохраняет и проверяет закрытую запись в архиве, затем удаляет её из `TODO.md`.

Если нужна отдельная будущая проверка результата, создать отдельную задачу проверки.

## 22. Границы проектирования

- Создавать ровно один project skill `manage-project-tasks`; не создавать конкурирующие task skills или отдельный task protocol.
- Не использовать `TODO.md` как архив, transcript или свалку идей.
- Не хранить длинные планы и объяснения внутри активной задачи.
- Не создавать дополнительные state/progress/handoff-файлы без реальной необходимости.
- Не вводить новые статусы, владельцев, dependency-графы или канбан-слои без явного запроса.
- Не дублировать task-политику в нескольких местах `AGENTS.md`.
- Не возвращать полную рабочую процедуру или точные шаблоны в корневой `AGENTS.md`.
- Не читать все архивы перед каждой задачей.
- Не выполнять одинаковую проверку несколько раз без ошибки или нового изменения.
- Не удалять закрытую задачу до успешной записи и проверки архива.
- Не переписывать нерелевантные задачи ради форматирования.

## 23. Сводка поведения установки для Codex

Когда этот документ передан с запросом «установи»:

1. Определить корень проекта.
2. Найти существующий task-файл, определить регистр имени и привести канонический файл к точному `TODO.md` без потери данных.
3. Определить действующий путь task-архива; для нового проекта использовать `archive/tasks/YYYY-MM.md`.
4. Полностью прочитать активный глобальный `skill-creator`.
5. Одним inspect-pass прочитать проектный `AGENTS.md`, существующий `manage-project-tasks`, `TODO.md` и только нужные сведения об архиве.
6. Зафиксировать временный снимок открытых задач, статусов, дат, proof и project/area-привязки.
7. Классифицировать состояние как новую, старую AGENTS-only, совместимую или конфликтующую установку.
8. Если требуется заметная миграция, один раз показать варианты, объяснить последствия и получить выбор пользователя.
9. Не перетирать адаптированную или конфликтующую систему без согласованного плана.
10. Создать или обновить один канонический project skill `manage-project-tasks`.
11. Встроить один короткий корневой контракт с фактическим путём архива и без подробных шаблонов.
12. Создать или совместимо обновить `TODO.md` как реестр открытой работы.
13. Перенести закрытые задачи в действующий архив только в рамках выбранного пользователем варианта.
14. Не удалять и не переименовывать прежние самостоятельные task-файлы или архивы, кроме безопасной нормализации регистра канонического `TODO.md`.
15. Не создавать пустые или лишние постоянные рабочие файлы.
16. Выполнить один setup-batch.
17. Проверить skill, установку и сохранность task-состояния как отдельные доказательные слои.
18. Если среда позволяет, выполнить изолированный behavior smoke-test на временной копии.
19. Отдельно указать, запускался ли fresh-context routing test.
20. Кратко сообщить, какие task-файлы изменены и какие оставлены без изменений.
21. Сообщить только доказанные уровни подтверждения.

После установки будущие сессии должны вести систему, используя проектный
`AGENTS.md`, project skill `manage-project-tasks`, `TODO.md` и нужный месячный
архив без повторного чтения этого документа. Простые справочные чтения
выполняются напрямую; `manage-project-tasks` подключается для записи,
жизненного цикла и восстановления долговременного task state.
